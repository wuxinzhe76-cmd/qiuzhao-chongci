## 八、注意事项

### 8.1 环境限制注意事项

| # | 注意事项 | 说明 |
|---|---------|------|
| 1 | 单机 4C16G 资源有限 | 并发压测不要超过 100，观察 CPU/内存使用率，避免 OOM |
| 2 | 中间件全部单机部署 | 无高可用测试条件，异常恢复测试仅模拟单点故障 |
| 3 | LLM 是 MiniMax-M3 | 非通义千问，延迟和输出格式可能不同，预期指标需灵活调整 |
| 4 | Rerank 是 DashScope gte-rerank API | 非本地 bge-reranker，延迟受网络影响，无法测试 Bi-Encoder-Rerank/LLM-Rerank 消融 |
| 5 | 消息队列是 RabbitMQ | 非 Kafka，记忆升级消息走 RabbitMQ，测试时注意队列名和消费者状态 |
| 6 | 无 Kafka/ClickHouse | 可观测性完整链路跳过，仅测 SDK 采集部分（如有） |
| 7 | 无 GraphRAG | GraphRAG 全部测试跳过 |
| 8 | Milvus 1047 条 / ES 217 条 | 数据量较小，测试集规模调整为 30 条 query（原计划 100 条） |
| 9 | 应用通过 Nginx 端口 80 访问 | 所有 API 调用走 `http://localhost:80/api/...` |

### 8.2 测试操作注意事项

| # | 注意事项 | 说明 |
|---|---------|------|
| 1 | 消融实验需重启应用 | 每次切换配置后重启应用，等待 10s 确认启动完成再测试 |
| 2 | 测试前清空缓存 | 语义缓存测试前清空 Redis，避免历史缓存干扰 |
| 3 | 记录测试环境快照 | 每次测试前记录 Docker 容器状态和资源使用率 |
| 4 | API 调用注意 QPS 限制 | MiniMax-M3 和 DashScope API 有 QPS 限制，批量测试注意限流 |
| 5 | 日志收集 | 测试期间开启应用日志收集，便于事后分析异常 |
| 6 | 数据可复现 | 每个 query 的检索结果需记录完整返回 JSON，便于复现 |
