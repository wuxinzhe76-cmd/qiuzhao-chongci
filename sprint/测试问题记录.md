## 问题 1：DashScope Rerank 模型名错误（已修复）

| 项目 | 内容 |
|------|------|
| **问题描述** | 测试脚本中 Rerank 模型名写的 `gte-rerank`，实际应为 `gte-rerank-v2` |
| **错误信息** | `AccessDenied: Access denied` |
| **影响** | 第一轮测试所有 Rerank 配置均降级为无 Rerank |
| **修复** | 已将模型名改为 `gte-rerank-v2`，验证 API 调用成功 |
| **代码影响** | 需检查 `interview-arena` 后端代码中 Rerank 模型名是否也写错了 |

## 问题 2：Milvus 与 ES 文档 ID 不匹配（架构问题）

| 项目 | 内容 |
|------|------|
| **问题描述** | Milvus 使用 UUID 作为 doc_id（如 `016fa218-5667-45e1-8b90-be7a6dddb1b4`），ES 使用 MySQL 数字 ID（如 `1`, `2`） |
| **影响** | BM25-Only 检索结果无法与 Milvus 文档匹配，消融实验中 BM25-Only 显示 0% 召回率 |
| **根因** | RAG 入库时 Milvus 和 ES 使用了不同的 ID 体系 |
| **建议修复** | 统一 ID 体系：Milvus 的 doc_id 应与 ES 的文档 ID 一致（都使用 MySQL question.id） |
| **临时方案** | 在测试脚本中将 ES 返回的数字 ID 映射到 Milvus 的 UUID（需要维护映射表） |

## 问题 3：RAG 测试集设计问题（测试设计）

| 项目 | 内容 |
|------|------|
| **问题描述** | 测试 query 直接使用文档标题（如"Redis 的订阅发布功能是什么？"），导致向量检索 100% 命中 |
| **影响** | 所有含向量检索的配置都显示 100% Hit@5，无法区分各模块贡献 |
| **根因** | query 与文档标题完全一致，向量检索和 BM25 都能精确匹配 |
| **建议修复** | 使用改写 query（同义改写、关键词提取、术语错误等）作为测试集 |
| **改进方案** | 1. 同义改写：原"Redis 的订阅发布功能是什么" -> 改"Redis Pub/Sub 机制" |
| | 2. 关键词查询：原"Redis 实现分布式锁时可能遇到的问题" -> 改"Redis 分布式锁 问题" |
| | 3. 术语错误：原"Redis 的哨兵机制" -> 改"Redis 的 sentinel 机制" |

## 问题 4：应用 API Token 验证失败（待排查）

| 项目 | 内容 |
|------|------|
| **问题描述** | 登录成功获取 accessToken，但调用 API 时返回"token 无效或已过期" |
| **已排查** | 1. Redis 中 `access:3` key 存在且值与 token 前缀匹配 |
| | 2. 尝试了 `Authorization: Bearer <token>`、`Authorization: <token>`、`token: <token>` 三种格式 |
| | 3. `Authorization: Bearer` 返回 40100（token 无效），`Authorization:` 无 Bearer 返回 500（系统异常） |
| **可能原因** | 1. JWT 签名密钥不匹配（应用可能使用了环境变量中的密钥） |
| | 2. Redis 中的 token 值与返回给客户端的 token 不完全一致 |
| | 3. JwtInterceptor 的验证逻辑有 bug |
| **影响** | 无法通过 API 测试 RAG 端到端回答质量、Agent 面试流程、安全测试 |
| **建议** | 检查 `JwtUtil.java` 和 `JwtInterceptor.java` 的 token 生成和验证逻辑 |

## 问题 5：MySQL 表结构与 Blueprint 不一致

| 项目 | 内容 |
|------|------|
| **问题描述** | Blueprint 中 user 表字段为 `userAccount`/`userPassword`，实际数据库为 `username`/`password` |
| **影响** | 登录 API 使用 `userAccount` 字段时返回"用户名不能为空" |
| **修复** | 使用 `username`/`password` 字段登录成功 |
| **代码影响** | 需更新 Blueprint 文档中的数据库设计部分 |

## 问题 7：SSH 命令转义问题（工具问题）

| 项目 | 内容 |
|------|------|
| **问题描述** | 通过 SSH 执行包含 JSON 的命令时，shell 转义导致 JSON 被破坏 |
| **影响** | curl 命令的 JSON body 经常被 mangled |
| **临时方案** | 使用 Python `json.dump()` 写入文件，再用 `curl -d @file` 读取 |
| **建议** | 复杂命令使用 Python 脚本而非 shell 命令 |
