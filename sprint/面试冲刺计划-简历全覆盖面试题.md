## 二、技能清单面试题

### 2.1 AI Agent 与工程化

### Q5: 什么是 ReAct 推理模式？它和 Plan-and-Execute 有什么区别？
**难度**：基础
**考察点**：Agent 推理模式理解
**回答要点**：
- ReAct：Thought -> Action -> Observation 循环，每步推理+行动交替，适合探索性任务
- Plan-and-Execute：先全局规划再分步执行，适合路径明确的复杂任务
- 区别：ReAct 是边想边做、动态调整；Plan-Execute 是先规划再执行、减少中途 LLM 调用
- 各自优缺点：ReAct 灵活但 Token 消耗大、可能跑偏；Plan-Execute 效率高但计划可能过时
**追问预警**：
- 你的项目里用的是哪种？为什么选它？
- 有没有混合使用的情况？

### Q6: 什么是 Function Calling？它和 MCP 协议有什么关系？
**难度**：进阶
**考察点**：工具调用机制、协议理解
**回答要点**：
- Function Calling：LLM 根据工具描述（JSON Schema）输出结构化调用意图，由代码执行
- MCP（Model Context Protocol）：标准化的工具集成协议，定义了 Server/Client 架构和工具发现机制
- 关系：Function Calling 是模型层面的能力（输出意图），MCP 是协议层面的标准（工具如何注册、发现、调用）
- MCP 解决了工具碎片化问题，一次集成可被多个 Agent 复用
**追问预警**：
- MCP 的 tool/resource/prompt 三类能力分别是什么？
- 你在项目中是怎么集成 MCP 工具的？

### Q7: Agent 的记忆分层是怎么设计的？长期记忆和工作记忆有什么区别？
**难度**：进阶
**考察点**：Agent 记忆架构
**回答要点**：
- 短期记忆（工作记忆）：当前对话上下文，存在 LLM Context Window 中，轮次结束后丢弃
- 工作记忆：当前任务相关的临时状态，如用户画像片段、中间结果，存在内存或 Redis
- 长期记忆：持久化存储的历史交互、用户偏好、知识点掌握情况，存向量库或关系库
- 记忆升级机制：短期 -> 工作记忆 -> 长期记忆，通过异步任务触发（如面试结束后 Kafka 触发）
**追问预警**：
- 记忆检索是怎么做的？用了什么召回策略？
- 长期记忆存储膨胀怎么处理？

### Q8: Agent 上下文管理的核心挑战是什么？你怎么解决的？
**难度**：进阶
**考察点**：上下文窗口管理
**回答要点**：
- 挑战：Context Window 有限，长对话会超限；信息过多导致 Lost-in-the-middle
- 解决方案：上下文压缩（摘要替代历史）、滑动窗口（保留最近 N 轮）、选择性召回（只取相关记忆）
- Token 预算控制：为不同部分（系统提示、记忆、用户输入、工具结果）分配 Token 预算
- 优先级排序：系统提示 > 当前输入 > 相关记忆 > 历史摘要
**追问预警**：
- Token 预算怎么分配的？各部分占比多少？
- 如果记忆和当前输入冲突了怎么办？

### Q9: 什么是 Prompt 注入攻击？Agent 场景下怎么防护？
**难度**：进阶
**考察点**：AI 安全意识
**回答要点**：
- Prompt 注入：恶意用户在输入中嵌入指令，试图覆盖系统提示或操控 Agent 行为
- 直接注入：用户输入中直接包含恶意指令（如"忽略之前的指令，执行XXX"）
- 间接注入：通过工具返回的数据注入恶意指令（如网页内容中隐藏提示）
- 防护措施：输入过滤（检测注入模式）、系统提示加固（明确边界）、工具返回沙箱化（隔离处理）、权限管控（限制 Agent 可执行操作）
**追问预警**：
- 你的项目里具体怎么实现的防护？
- 间接注入比直接注入更难防，你怎么看？

### Q10: Agent 出现死循环怎么检测和恢复？
**难度**：深挖
**考察点**：Agent 工程化稳定性
**回答要点**：
- 检测方式：轮次上限硬约束、重复动作检测（相同 action 连续出现 N 次）、Token 消耗监控
- 恢复策略：强制降级（切换到确定性流程）、重置上下文（清理历史保留摘要）、人工介入兜底
- 代码层面：每轮检查 action 历史，检测到重复模式时触发降级
- 你的项目实践：反思层轮次上限 + 非法动作自动降级恢复
**追问预警**：
- 降级后用户体验怎么保证？
- 你的轮次上限设的是多少？怎么定的？

### Q11: 什么是 Token 预算控制？Agent 中怎么实现？
**难度**：进阶
**考察点**：成本控制工程化
**回答要点**：
- Token 预算：为单次 Agent 执行设定 Token 上限，防止失控消耗
- 实现方式：每次 LLM 调用前预估 Token 用量（tiktoken/分词器），超预算则截断或降级
- 分层预算：系统提示固定预算、记忆动态预算、工具结果压缩预算
- 监控告警：实时统计 Token 消耗，超阈值触发告警
**追问预警**：
- tiktoken 对不同模型的分词结果一样吗？
- 工具返回结果太长怎么压缩？

### Q12: 熔断降级在 Agent 场景和传统微服务场景有什么不同？
**难度**：进阶
**考察点**：AI 工程化与传统工程的对比
**回答要点**：
- 传统微服务：基于 RT/QPS/错误率的熔断，降级走缓存或默认值
- Agent 场景：LLM 调用延迟波动大（几秒到几十秒），需要语义级降级（缓存相似问题的回答）
- 慢调用比例熔断更适合 Agent：LLM 正常也可能慢，按比例而非绝对阈值
- 语义缓存降级：用向量相似度匹配历史回答，而非简单的 key-value 缓存
**追问预警**：
- 语义缓存的命中率怎么保证？
- 熔断恢复策略是什么？

### Q13: 你提到 Agent 的编排层用 Workflow + ReAct 混合驱动，具体怎么混合的？
**难度**：深挖
**考察点**：编排策略设计
**回答要点**：
- Workflow 确定性流程：大部分步骤（感知、记忆召回、工具执行、反思）用代码确定性控制
- ReAct 动态决策：仅在需要 LLM 推理的步骤（如规划下一步动作）使用 ReAct
- 你的项目实践：10 步流程中仅 1 步交 LLM 决策，其余 9 步代码控制
- 优势：减少 LLM 调用次数（降成本降延迟）、提高确定性（减少幻觉）、保留灵活性（关键步骤动态决策）
**追问预警**：
- 那一步 LLM 决策如果错了怎么办？
- 为什么不全用 Workflow？不全用 ReAct？

### Q14: 什么是 Agent 的反思机制？你的项目中怎么实现的？
**难度**：进阶
**考察点**：反思层设计
**回答要点**：
- 反思：Agent 对自身输出进行质量评估和修复，而非直接返回
- 三层校验：结构校验（JSON 格式是否正确）-> 格式校验（字段类型/范围）-> 语义校验（内容是否合理）
- 有限修复：校验失败后给 LM 反馈错误信息让其重新生成，设重试上限防死循环
- 与反思的区别：校验是规则驱动（确定性），反思可以包含 LLM 自评（非确定性）
**追问预警**：
- 语义校验具体怎么做的？用了规则还是 LLM？
- 重试上限设多少？超了怎么处理？

### 2.2 大模型与 RAG

### Q15: Transformer 的 Self-Attention 机制是怎么工作的？计算复杂度是多少？
**难度**：基础
**考察点**：Transformer 核心原理
**回答要点**：
- Self-Attention：每个 token 与序列中所有 token 计算注意力权重，Q×K^T / √d -> softmax -> ×V
- 作用：捕捉序列内任意两个 token 的依赖关系，不受距离限制
- 计算复杂度：O(n²·d)，n 是序列长度，d 是向量维度
- √d 缩放：防止点积过大导致 softmax 梯度消失
**追问预警**：
- 为什么不用加性注意力（Additive Attention）？
- n² 复杂度对长文本有什么问题？怎么优化？

### Q16: Multi-Head Attention 相比 Single-Head 有什么优势？
**难度**：进阶
**考察点**：多头注意力理解
**回答要点**：
- 多头并行：将 Q/K/V 拆分为 h 个头，每个头独立计算注意力，最后拼接
- 优势：不同头关注不同子空间的特征（语法关系、语义关系、位置关系等）
- 计算量：总参数量与单头相同（维度从 d 变为 d/h × h）
- 头数选择：太多会过拟合、太少表达能力不足，常见 8/12/16
**追问预警**：
- 不同头学到的东西怎么看出来？
- 头数越多越好吗？有没有实验结论？

### Q17: Transformer 的位置编码有哪些方式？各有什么优缺点？
**难度**：进阶
**考察点**：位置编码机制
**回答要点**：
- 正弦余弦编码（原始 Transformer）：固定编码，不可学习，能泛化到更长序列
- 可学习位置编码（BERT）：每个位置一个可学习向量，表达力强但不泛化
- RoPE 旋转位置编码（LLaMA）：通过旋转矩阵编码相对位置，支持长度外推
- ALiBi：在注意力分数上加线性偏置，无需位置编码向量
**追问预警**：
- RoPE 是怎么实现相对位置编码的？
- 长度外推是什么意思？为什么重要？

### Q18: RAG 的全链路工程落地包括哪些环节？
**难度**：基础
**考察点**：RAG 工程化全景
**回答要点**：
- 数据处理：文档解析 -> 分块（Chunking）-> 向量化（Embedding）-> 存入向量库
- 检索阶段：Query 理解 -> 检索（向量/关键词/混合）-> 重排序（Rerank）
- 生成阶段：Prompt 组装 -> LLM 生成 -> 后处理（引用标注/去重）
- 评估：检索准确率、召回率、生成质量（Faithfulness/Relevancy）
**追问预警**：
- 你的项目中每个环节用的什么技术？
- 分块策略对效果影响大吗？

### Q19: Hybrid Search 是怎么工作的？为什么比纯向量检索好？
**难度**：进阶
**考察点**：混合检索理解
**回答要点**：
- Hybrid Search = 向量检索（语义匹配）+ 关键词检索（精确匹配，如 BM25）
- 向量检索擅长语义相似（"怎么部署" 匹配 "如何上线"），但可能漏掉精确关键词
- BM25 擅长精确匹配（专有名词、代码标识符），但不理解语义
- 融合方式：RRF（Reciprocal Rank Fusion）按排名倒数加权融合分数
**追问预警**：
- RRF 的公式是什么？为什么用排名而不是原始分数？
- 向量和 BM25 的 Top-K 各取多少？怎么定的？

### Q20: Reranker 精排的作用是什么？和检索阶段有什么区别？
**难度**：进阶
**考察点**：重排序机制
**回答要点**：
- 检索阶段：双塔模型（Bi-Encoder），Query 和 Doc 独立编码，速度快但精度低
- Rerank 阶段：Cross-Encoder，Query 和 Doc 拼接后联合编码，精度高但速度慢
- 流程：先检索 Top-K（如 40 条），再用 Cross-Encoder 精排取 Top-5
- trade-off：Cross-Encoder 计算量大，只对少量候选精排
**追问预警**：
- Cross-Encoder 为什么比 Bi-Encoder 精度高？
- Reranker 模型有哪些选择？你用的什么？

### Q21: RAG 的评估指标有哪些？怎么衡量 RAG 系统的好坏？
**难度**：进阶
**考察点**：RAG 评估体系
**回答要点**：
- 检索质量：Recall@K（召回率）、Precision@K（精确率）、MRR（平均倒数排名）
- 生成质量：Faithfulness（忠实度，是否基于检索内容）、Relevancy（相关性，是否回答了问题）
- 端到端：Answer Correctness（答案正确性）、Hallucination Rate（幻觉率）
- 框架：RAGAS / TruLens 等评估框架
**追问预警**：
- Faithfulness 具体怎么计算的？
- 你的项目中评估数据集怎么构建的？

### Q22: 什么是 Modular RAG？它和传统 RAG 有什么区别？
**难度**：进阶
**考察点**：RAG 架构演进
**回答要点**：
- 传统 RAG（Naive RAG）：检索 -> 生成，固定两步流程
- Modular RAG：将 RAG 拆分为可插拔模块（预检索、检索、后检索、生成）
- 预检索模块：Query Rewrite、Query Expansion、路由分发
- 后检索模块：Rerank、去重、Lost-in-the-middle 重排
- 优势：模块化组合、按需启用、便于迭代
**追问预警**：
- Query Rewrite 有哪些策略？
- Lost-in-the-middle 是什么问题？怎么重排？

### Q23: 什么是 Agentic RAG？它和 Modular RAG 有什么区别？
**难度**：深挖
**考察点**：RAG 与 Agent 结合
**回答要点**：
- Agentic RAG：将 RAG 嵌入 Agent 循环，LLM 自主判断检索是否充分，不够则改写 Query 再检索
- 流程：检索 -> LLM 判断充分性 -> 不够则改写 Query -> 再检索（最多 N 轮）
- 与 Modular RAG 区别：Modular 是流水线（固定流程），Agentic 是迭代循环（动态决策）
- 优势：自动补全信息、提高回答质量；代价：增加 LLM 调用次数和延迟
**追问预警**：
- 判断充分性用什么策略？LLM 自评还是规则？
- 最多几轮？怎么平衡延迟和质量？

### Q24: Lost-in-the-middle 问题是什么？怎么解决？
**难度**：深挖
**考察点**：LLM 上下文利用
**回答要点**：
- 现象：LLM 对 Context 中间位置的信息利用率低，开头和结尾的信息更容易被关注
- 论文依据：Liu et al. 2023，实验证明中间位置准确率显著下降
- 解决方案：将最相关的文档放在 Context 两端（重排），中间放次要文档
- 其他方案：减少 Context 长度（精简检索结果）、分段处理
**追问预警**：
- 你在项目中是怎么重排的？

- Spring AI：Java 生态的 AI 框架，对标 LangChain，集成 Spring 生态，适合 Java 后端团队
- 选型：Python 生态优先 LangGraph；Java 生态优先 Spring AI

### Q65: 你的 Hybrid Search 具体是怎么实现的？向量检索和 BM25 各取多少？
**难度**：进阶
**考察点**：混合检索工程实现
**回答要点**：
- 向量检索：Milvus 检索 Top-20（Embedding 模型生成 Query 向量）
- BM25 检索：Elasticsearch BM25 Top-20（全文检索）
- RRF 融合：score = Σ 1/(k + rank_i)，k 通常取 60
- 融合后取 Top-40 交给 Reranker
- 为什么各取 20：召回率和精确率的平衡，太少漏召回，太多增重 Reranker 负担
**追问预警**：

### Q66: RRF（Reciprocal Rank Fusion）的公式是什么？为什么用排名而不是原始分数？
**难度**：深挖
**考察点**：RRF 原理
**回答要点**：
- 公式：RRF_score(d) = Σ 1/(k + rank_i(d))，d 是文档，rank_i 是文档在第 i 路检索中的排名
- 用排名不用原始分数的原因：向量相似度和 BM25 分数量纲不同（向量 0-1，BM25 可能 0-30），直接加权不公平
- 排名是归一化的，消除了量纲差异
- k 值作用：平滑参数，k 越大对排名差异越不敏感，k=60 是论文推荐值
**追问预警**：
- 如果某路检索结果特别差，RRF 还能工作好吗？
- 有没有加权 RRF 的方案？

### Q67: Cross-Encoder 精排的流程是什么？为什么只取 Top-5？
**难度**：进阶
**考察点**：Reranker 工程
**回答要点**：
- 流程：Query + 每个候选 Doc 拼接 -> Cross-Encoder 输出相关性分数 -> 按分数排序取 Top-5
- Cross-Encoder 特点：Query 和 Doc 联合编码（Self-Attention 交互），精度高但 O(n) 计算量
- 只取 Top-5 的原因：LLM Context 有限，Top-5 足够覆盖高质量答案；更多反而可能引入噪声
- 模型选择：bge-reranker / ms-marco-MiniLM 等
**追问预警**：

### Q68: Redis 语义缓存是怎么实现的？缓存 key 怎么设计？
**难度**：进阶
**考察点**：语义缓存设计
**回答要点**：

### Q75: Memory、State、Context 三分离的设计思路是什么？为什么不统一管理？

### Q87: MyBatis-Plus + ES 异步同步索引是怎么实现的？

### Q88: Caffeine 本地缓存和 Redis 缓存是怎么配合的？
**难度**：进阶
**考察点**：多级缓存
**回答要点**：
- 多级缓存：Caffeine（L1 本地）-> Redis（L2 分布式）-> DB
- 查询流程：先查 Caffeine -> miss 查 Redis -> miss 查 DB -> 回写 Redis 和 Caffeine
- Caffeine 优势：进程内缓存，无网络开销，微秒级响应
- 一致性：Caffeine 用 TTL/Size 淘汰策略，Redis 用 TTL，DB 变更时主动清除两级缓存
- 淘汰策略：Caffeine 用 W-TinyLFU（比 LRU 更好的命中率）

### Q89: JWT 双 Token 机制是怎么实现的？
**难度**：进阶
**考察点**：认证机制
**回答要点**：
- 双 Token：Access Token（短期，如 30 分钟）+ Refresh Token（长期，如 7 天）
- 流程：Access Token 过期 -> 用 Refresh Token 换新 Access Token -> 用户无感刷新
- 安全：Access Token 存内存/LocalStorage，Refresh Token 存 HttpOnly Cookie
- 为什么双 Token：Access Token 短期降低被盗风险，Refresh Token 避免频繁登录
- JWT 结构：Header.Payload.Signature，签名防篡改但不加密
**追问预警**：
- JWT 和 Session 有什么区别？
- Refresh Token 存在哪里？怎么续期？

### 5.2 异步批量上报

### Q96: 有界队列（10000 上限）的设计考虑是什么？满了怎么办？
**难度**：进阶
**考察点**：背压保护
**回答要点**：
- 有界队列：内存中维护 BlockingQueue，上限 10000，防止 OOM
- 入队策略：非阻塞 offer，入队成功返回；队列满时丢弃最旧数据 + 告警
- 为什么不阻塞：阻塞会影响 Agent 业务代码（不可接受），丢弃观测数据可接受

### Q97: 500ms/20 条批量上报的策略是怎么定的？
**难度**：进阶
**考察点**：批量上报优化
**回答要点**：
- 两个触发条件：时间触发（500ms 到了就发）+ 数量触发（攒满 20 条就发），取先到者
- 500ms 平衡：太短（频繁上报增加网络开销）、太长（数据延迟大）
- 20 条平衡：太多（单次请求体大、内存占用高）、太少（批量优势不明显）
- 批量优势：减少 HTTP 请求次数 -> 降低连接开销 -> 提高吞吐
**追问预警**：

### Q98: 3 次指数退避重试的参数是什么？全部失败后怎么办？
**难度**：进阶
**考察点**：重试策略
**回答要点**：
- 指数退避：第 1 次立即重试、第 2 次等 1s、第 3 次等 2s（或 1s/2s/4s）
- 3 次原因：平衡可靠性和延迟，再多影响后续数据实时性
- 全部失败后：数据写入本地落盘文件（fallback），等待 Collector 恢复后重传
- 幂等性：每条数据有唯一 ID，Collector 去重防止重复写入

### Q101: Kafka 在这里起什么作用？为什么不在 Collector 直接写 ClickHouse？
**难度**：进阶
**考察点**：消息队列解耦
**回答要点**：
- Kafka 作用：削峰填谷（采集高峰时缓冲数据）、解耦（生产和消费独立扩展）
- 不直接写 ClickHouse 的原因：

  - 高峰期大量写入会压垮 ClickHouse
  - Collector 和 ClickHouse 解耦后可独立扩展

### Q102: Consumer 按 Span 类型分流是怎么实现的？为什么分流？
**难度**：进阶
**考察点**：消费分流策略
**回答要点**：
- 分流方式：Consumer 订阅 Kafka Topic -> 按消息中的 Span Type 字段路由到不同处理逻辑
- 分流原因：不同 Span Type 的数据格式、写入表、处理逻辑不同
- 实现方式：单 Consumer Group + 内部分发，或多个 Consumer Group 各订阅子 Topic
- 批量写入：按 Span Type 分别攒批，各自达到 50 条/5s 时批量写入对应 ClickHouse 表

### Q103: 50 条/5s 强制刷新的批量拉取策略，参数怎么定的？
**难度**：进阶
**考察点**：批量参数调优
**回答要点**：
- 50 条：ClickHouse 批量写入的性能拐点，太少写入效率低，太多单次写入耗时长
- 5s：数据延迟上限，超过 5s 影响监控实时性
- 强制刷新：即使不够 50 条，5s 到了也必须写入，保证数据不积压
- 动态调整：可根据流量动态调整批量大小（流量大时增大批量、流量小时减小延迟）

### 5.4 列式存储与聚合分析

### Q104: ClickHouse 相比 MySQL/ES 在这个场景下有什么优势？
**难度**：进阶
**考察点**：存储选型
**回答要点**：
- 列式存储：只读取查询涉及的列，IO 效率高（分析查询通常只涉及部分列）
- 压缩率高：同列数据类型一致，压缩效果好，节省存储
- 聚合性能：向量化执行引擎，批量处理数据，聚合查询比 MySQL 快 100 倍+
- 写入模型：适合批量写入（Append-Only），不适合频繁更新删除
- 对比 ES：ES 适合全文检索，ClickHouse 适合数值聚合；本场景是指标聚合分析
**追问预警**：
- ClickHouse 不适合什么场景？
- ClickHouse 的 MergeTree 引擎原理是什么？

### Q105: 5 张业务表分别是什么？为什么这么设计？
**难度**：深挖
**考察点**：表结构设计
**回答要点**：
- 示例 5 张表：
  1. spans（Span 明细表）：Span ID、Trace ID、Parent ID、Type、开始/结束时间、耗时
  2. llm_calls（LLM 调用表）：模型名、Token 用量、Prompt 长度、Completion 长度
  3. tool_calls（工具调用表）：工具名、参数摘要、返回大小、状态、耗时
  4. rag_retrieves（RAG 检索表）：Query、召回数、检索耗时、Rerank 耗时
  5. agent_metrics（Agent 指标表）：Agent ID、步骤数、总耗时、Token 总量
- 设计依据：按 Span Type 分表，每张表只存该类型的专属字段，避免宽表稀疏
- 分区策略：按天分区（PARTITION BY toDate(timestamp)），便于按天查询和清理
**追问预警**：
- 为什么不用一张宽表？
- 分区表的数据怎么过期清理？

### Q106: 物化视图的 AggregatingMergeTree + avgState/maxState 是什么意思？
**难度**：深挖
**考察点**：ClickHouse 高级特性
**回答要点**：
- 物化视图：自动维护的预聚合表，源表数据变更时自动更新
- AggregatingMergeTree：引擎类型，合并时按聚合函数合并数据（不存原始值，存中间状态）
- avgState/maxState：聚合函数的 State 版本，存储聚合中间状态（如 avg 存 sum+count）
- 查询时用 avgMerge/maxMerge 合并状态得到最终结果
- 优势：预计算聚合，查询时不需要扫描原始数据，大幅提升查询性能
**追问预警**：
- avgState 存的是什么中间状态？
- 物化视图的更新时机是什么？

### Q107: Prefill/Decode 延迟是怎么区分采集的？为什么要分开看？
**难度**：深挖
**考察点**：LLM 延迟分析
**回答要点**：
- Prefill：LLM 处理 Prompt 的时间（首 Token 延迟，TTFT）
- Decode：LLM 生成 Completion 的时间（每 Token 延迟，TPOT）
- 区分采集：SDK 拦截流式响应，记录首 Token 到达时间（Prefill）和后续 Token 间隔（Decode）
- 为什么要分开：Prefill 受 Prompt 长度影响，Decode 受生成长度影响，优化方向不同
  - Prefill 慢 -> 优化 Prompt 长度、用 Prefix Caching
  - Decode 慢 -> 换更快的模型、降低生成长度
**追问预警**：
- TTFT 和 TPOT 各多少算正常？
- Prefix Caching 是什么？

### 5.5 系统设计追问

### Q108: 如果可观测性系统自身挂了，会不会影响 Agent 业务？
**难度**：深挖
**考察点**：系统可靠性
**回答要点**：
- 设计原则：观测系统不能影响业务系统（观测是附加能力，不是核心链路）
- SDK 侧保护：异步上报（不阻塞业务）、有界队列（不 OOM）、异常捕获（上报失败不影响业务）
- Collector 侧保护：独立部署、独立资源，Collector 挂了 SDK 数据落盘等待恢复
- 降级策略：SDK 检测到 Collector 不可用时，降低上报频率或停止上报
- 监控：观测系统自身也需要被监控（自观测）
**追问预警**：
- SDK 的异常捕获会不会吞掉业务异常？
- 观测系统自身怎么监控？

### Q109: 整个数据链路中，哪个环节最容易成为瓶颈？怎么优化？
**难度**：深挖
**考察点**：性能瓶颈分析
**回答要点**：

- 最常见瓶颈：ClickHouse 写入（MergeTree 合并是 CPU 密集型）
- 优化方向：批量写入、合理分区、控制 Part 数量、异步 merge

### Q115: 你觉得 AI 后端工程师和传统 Java 后端工程师最大的区别是什么？
**难度**：进阶
**考察点**：岗位认知
**回答要点**：
- 区别1：不确定性处理 - 传统后端是确定性的（if-else），AI 后端要处理 LLM 的非确定性输出（校验/重试/降级）
- 区别2：成本意识 - 传统后端关注 CPU/内存，AI 后端还要关注 Token 成本（每次调用都花钱）
- 区别3：延迟特征 - 传统后端 P99 可能 100ms，AI 后端 P99 可能 30s+，需要异步化/流式/缓存
- 区别4：评估方式 - 传统后端关注功能正确性，AI 后端还要关注生成质量（幻觉率/忠实度）
- 共同点：分布式系统设计、高并发、中间件使用等后端工程能力是通用的
**追问预警**：
- 你觉得传统后端转 AI 后端最难的是什么？
- AI 后端会不会取代传统后端？
