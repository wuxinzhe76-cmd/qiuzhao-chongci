### 1.4 评估指标体系

#### 1.4.1 检索质量指标

| 指标 | 定义 | 计算公式 | 目标值 |
|------|------|---------|--------|
| Hit Rate@K | 前 K 个结果中包含正确答案的比例 | 命中查询数 / 总查询数 | Top-5 ≥ 90% |
| MRR (Mean Reciprocal Rank) | 正确答案排名倒数的均值 | avg(1 / 正确答案排名) | ≥ 0.72 |
| Recall@K | 前 K 个结果覆盖所有相关文档的比例 | Σ(命中相关文档) / Σ(全部相关文档) | Top-20 ≥ 85% |
| Precision@K | 前 K 个结果中相关文档的比例 | 相关文档数 / K | Top-5 ≥ 0.75 |
| NDCG@K | 考虑排名位置的归一化折损累计增益 | DCG@K / IDCG@K | Top-5 ≥ 0.82 |

#### 1.4.2 Agent 质量指标

| 指标 | 定义 | 计算公式 | 目标值 |
|------|------|---------|--------|
| 任务完成率 | Agent 成功完成面试流程的比例 | 完成任务数 / 总任务数 | ≥ 85% |
| 动作准确率 | AI 输出 action_directive 正确的比例 | 正确动作数 / 总动作数 | ≥ 90% |
| 错误恢复率 | 发生错误后成功恢复的比例 | 恢复成功次数 / 错误总次数 | ≥ 95% |
| 轮次合理度 | 实际轮次落在合理区间的比例 | 合理轮次任务数 / 总任务数 | ≥ 80% |
| 幻觉率 | AI 生成不存在内容的比例 | 幻觉轮次数 / 总轮次数 | ≤ 5% |

#### 1.4.3 性能指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| P50 延迟 | 50% 请求的响应时间 | RAG ≤ 200ms，Agent ≤ 800ms |
| P95 延迟 | 95% 请求的响应时间 | RAG ≤ 500ms，Agent ≤ 2000ms |
| P99 延迟 | 99% 请求的响应时间 | RAG ≤ 1000ms，Agent ≤ 3000ms |
| QPS 吞吐 | 每秒处理请求数 | ≥ 100 QPS |
| 服务可用性 | 压测期间正常响应比例 | ≥ 99.9% |

#### 1.4.4 RAG 回答质量指标（RAGAS 框架）

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|---------|--------|
| Faithfulness | 回答是否忠于检索到的上下文 | LLM 判断回答中每个 claim 是否可被上下文支撑 | ≥ 0.85 |
| Answer Relevance | 回答与问题的相关程度 | LLM 从回答反向生成问题，计算与原问题的相似度 | ≥ 0.80 |
| Context Relevance | 检索上下文与问题的相关程度 | 上下文中相关句子占比 | ≥ 0.70 |
| Context Recall | 上下文覆盖标准答案的程度 | 上下文覆盖的标准答案句子数 / 标准答案总句子数 | ≥ 0.80 |

#### 2.1.2 评估指标定义

已在 1.4.1 节定义，此处补充检索阶段的计算细节：

```
Hit Rate@5 = (Top-5 结果中包含至少 1 个标准答案文档的 query 数) / 100
MRR = Σ(1 / rank_i) / 100，其中 rank_i 是第 i 条 query 正确答案的最高排名
NDCG@5 = DCG@5 / IDCG@5
  DCG@5 = Σ(rel_i / log2(i+1)) for i in 1..5
  rel_i = 1 if 文档 i 是标准答案 else 0
```

### 2.2 混合检索消融实验

#### 2.2.1 实验设计

**目标**：量化向量检索和 BM25 检索各自对混合召回的贡献，验证 RRF 融合的必要性。

**基线**：完整 Hybrid Search 链路（向量 Top-20 + BM25 Top-20 → RRF 融合 → Top-20）

**消融方法**：逐个去掉某一路检索，或去掉 RRF 融合改为简单拼接，对比检索质量指标变化。

#### 2.2.2 消融配置表

| 配置名 | 去掉的链路 | 保留的链路 | 融合方式 | 预期影响 |
|--------|-----------|-----------|---------|---------|
| Full（基线） | 无 | 向量 Top-20 + BM25 Top-20 | RRF 融合（k=60） | 最优，Hit@5 ≥ 90% |
| Vector-Only | BM25 检索 | 向量 Top-20 | 无 | 事实型 query 下降 5-8%，比较型下降 10-15% |
| BM25-Only | 向量检索 | BM25 Top-20 | 无 | 边界型 query（术语错误）下降 20%+，推理型下降 10% |
| No-RRF | RRF 融合 | 向量 + BM25 | 简单去重拼接（向量优先） | 排序质量下降，MRR 下降 0.1+ |
| RRF-k1 | 无 | 向量 + BM25 | RRF 融合（k=30） | 对高频词 query 略有提升，低频词下降 |
| RRF-k2 | 无 | 向量 + BM25 | RRF 融合（k=120） | 对低频词 query 略有提升，高频词下降 |

#### 2.2.3 测试 Prompt

以下 query 在各消融配置下分别执行，记录 Top-5/Top-10/Top-20 的命中情况：

```
# 事实型 query（预期 BM25 贡献大）
Q1: "Redis 的 RDB 持久化 save 配置规则"

# 语义型 query（预期向量贡献大）
Q3: "怎么防止 Agent 陷入无限循环"  # 标准答案用"死循环"而非"无限循环"
Q4: "记忆系统的遗忘机制"  # 标准答案用"memory consolidation"

# 混合型 query（预期 RRF 融合贡献大）
Q5: "Hybrid Search 的 RRF 融合算法公式"
Q6: "Cross-Encoder 精排和 Bi-Encoder 双塔编码器的区别"

# 边界型 query（预期 Query Rewrite + 向量贡献大）
Q7: "GVM 内存模型"  # 需纠正为 JVM
Q8: "agent 的 react 循环"  # 小写，需标准化
```

### 2.3 Rerank 精排消融实验

#### 2.3.1 实验设计

**目标**：量化 Cross-Encoder 精排对检索质量的边际贡献，以及不同 Rerank 策略的效果差异。

**基线**：Hybrid Search Top-20 -> DashScope gte-rerank API 精排 -> Top-5

#### 2.3.2 消融配置表

| 配置名 | 去掉的链路 | 保留的链路 | 预期影响 |
|--------|-----------|-----------|---------|
| Full-Rerank（基线） | 无 | Cross-Encoder 精排 Top-5 | 最优，Top-5 精确度 ≥ 0.75 |
| No-Rerank | Cross-Encoder 精排 | 直接取 RRF Top-5 | Top-5 精确度下降 15-20%，Lost-in-the-middle 问题加剧 |
| No-Rerank-Top10 | Cross-Encoder | 直接取 RRF Top-10 不精排 | 精确度下降，但召回率不变 |
| LLM-Rerank | Cross-Encoder | LLM 打分重排（MiniMax-M3） | 精确度接近 gte-rerank，但延迟增加 300%+ |
| Rerank-Top3 | 无 | Cross-Encoder 精排 Top-3 | 精确度略升，但召回率下降 |

#### 2.3.3 测试 Prompt

```
# 需要精排才能区分的 query（正确答案在 RRF Top-20 中排名靠后）
Q1: "Agent 的记忆系统怎么管理长期记忆"  # 检索结果中多个记忆相关文档，需精排区分
Q2: "ReAct 和 Plan-Execute 哪个更适合多步推理"  # 多个推理模式文档，需精排选最相关
Q3: "工具调用失败后的重试策略"  # 多个错误处理文档，需精排选工具层相关
```

#### 2.3.4 预期产出数据

| Full-Rerank | 0.78 | 0.78 | 0.85 | 420ms |
| No-Rerank | 0.62 | 0.65 | 0.72 | 180ms |
| No-Rerank-Top10 | 0.65 | 0.68 | 0.75 | 180ms |
| LLM-Rerank | 0.77 | 0.77 | 0.84 | 1200ms |

### 2.4 Modular RAG 三阶段消融实验

#### 2.4.1 实验设计

**目标**：量化预检索（Query Rewrite）、检索（Hybrid+RRF）、后检索（Rerank+去重+Lost-in-the-middle 重排）三阶段各自的贡献。

**基线**：完整 Modular RAG 三阶段链路

#### 2.4.2 消融配置表

| 配置名 | 预检索 | 检索 | 后检索 | 预期影响 |
|--------|--------|------|--------|---------|
| Full-Modular（基线） | ✅ Query Rewrite | ✅ Hybrid+RRF | ✅ DashScope gte-rerank+去重+重排 | 最优 |
| No-PreRetrieval | ❌ 去掉 Query Rewrite | ✅ Hybrid+RRF | ✅ DashScope gte-rerank+去重+重排 | 边界型 query（术语错误/短 query）下降 20%+ |
| No-PostRetrieval | ✅ Query Rewrite | ✅ Hybrid+RRF | ❌ 去掉 Rerank+去重+重排 | Top-5 精确度下降 15-20% |
| No-Both | ❌ 去掉 Query Rewrite | ✅ Hybrid+RRF | ❌ 去掉 Rerank+去重+重排 | 综合下降 25-30% |
| No-Dedup | ✅ Query Rewrite | ✅ Hybrid+RRF | 部分（Rerank+重排，无去重） | 重复文档导致 Top-5 有效结果减少 |
| No-Reorder | ✅ Query Rewrite | ✅ Hybrid+RRF | 部分（Rerank+去重，无 Lost-in-middle 重排） | 正确答案在中间位置时丢失率上升 |

#### 2.4.3 测试 Prompt

```

# 测试后检索去重的价值（向量+BM25 检索出重复文档）
Q4: "Kafka 消费者组 rebalance"  # 向量和 BM25 可能返回同一文档的不同 chunk
Q5: "Agent 工具调用沙箱化"  # 同一概念在多个文档中出现

# 测试 Lost-in-the-middle 重排的价值
Q6: "Hybrid Search RRF 融合"  # 正确答案可能在 Top-20 的第 10-15 位
```

### 2.5 Agentic RAG 迭代检索测试

#### 2.5.1 测试设计

**目标**：验证 Agentic RAG 的迭代检索（检索 → LLM 判断充分性 → 不够则改写 query 再检索，最多 3 轮）相比单次检索的提升。

**测试场景**：
1. **单跳问答**：一次检索即可回答（预期 1 轮即终止）
2. **多跳问答**：需要 2-3 轮检索才能拼凑完整答案
3. **信息不足**：3 轮检索后仍信息不足，应返回"信息不足"而非编造

#### 2.5.2 测试 Prompt

```
# 单跳问答（预期 1 轮）
Q1: "ReAct 模式的三个核心组件是什么？"

# 多跳问答（预期 2-3 轮）
Q2: "Agent 的记忆层如何与规划层协作？先检索记忆层架构，再检索规划层调用记忆的方式"
Q3: "Hybrid Search 的 RRF 融合公式是什么？这个公式中的 k 值一般设多少？为什么？"
   # 第 1 轮检索 RRF 公式，第 2 轮检索 k 值选择依据

# 信息不足场景（预期 3 轮后返回"信息不足"）
Q4: "2026 年最新的 Agent 评估基准测试有哪些？"  # 知识库中无此内容

# 迭代改写测试（第 1 轮 query 不够好，需 LLM 改写）
Q5: "Agent 怎么工作"  # 第 1 轮结果太泛，第 2 轮改写为 "Agent 6层Harness架构工作流程"
```

#### 2.5.3 预期产出数据

| 场景 | 平均迭代轮次 | Hit@5 | Faithfulness | P95 延迟 |
|------|------------|-------|-------------|---------|
| 单跳问答 | 1.0 | 92% | 0.88 | 420ms |
| 多跳问答 | 2.3 | 89% | 0.86 | 980ms |
| 信息不足 | 3.0（达到上限终止） | N/A | 0.95（正确返回"信息不足"） | 1500ms |
| 迭代改写 | 1.8 | 87% | 0.85 | 750ms |
