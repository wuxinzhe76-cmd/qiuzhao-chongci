### 2026-06-29 · main.py（FastAPI 入口）

**学了什么**：
- main.py 6 块拆解：导入 / 创建 app / CORS / 路由注册 / 生命周期 / 健康检查
- Router vs Controller（Spring 按类组织自动扫描，FastAPI 按 router 对象手动 include）
- CORS 跨域原理（浏览器同源策略 + 后端响应头告诉浏览器"我允许谁"）
- 前端 proxy vs 后端 CORS vs Nginx 反向代理（三种消除跨域的方式）
- 生命周期事件 startup/shutdown（= `@PostConstruct`/`@PreDestroy`，启动建 Kafka 连接，关闭释放）
- 装饰器本质（函数接收函数返回新函数，运行时自己执行）vs Java 注解（标记，需框架扫描）
- 装饰器 vs AOP（装饰器能实现 AOP 效果但不止干 AOP，还能注册事件/路由）
- 装饰器来源（框架自带 / Python 内置 / 第三方 / 自己写）

**关键认知**：
- 装饰器是"自己干活的函数"，Java 注解是"需要框架扫描的标签"
- `@app.on_event` 不是 AOP，是注册模式；`@app.get` 也不是 AOP，是路由注册
- shutdown 顺序跟 startup 相反：先停 consumer 再停 producer（先停消费方再停生产方）
- Spring 自动扫描 `@RestController`，FastAPI 没自动扫描要手动 `include_router`

**关键认知**：

### 2026-06-29 · kafka/producer.py + consumer.py（Kafka 生产者 + 消费者）

**学了什么**：
- Python 下划线命名习惯（_前缀=内部使用，不是强制 private）
- None = Java null（空值占位）
- global 声明（Python 函数内改全局变量必须显式声明，Java 不需要）
- init_producer：Kafka 配置参数（acks=1/max_batch_size=16384/linger_ms=10/compression=gzip）
- 序列化器注入机制（构造时传 value_serializer，send 内部自动调用，不是接口重写）
- async/await 异步机制（await 等的是内存操作，不等 Kafka 确认）
- Future 对象 + add_done_callback（= Java Future + Callback，任务完成后自动调）
- send_batch 非阻塞投递（用 send() 不用 send_and_wait()，不阻塞 FastAPI）
- send_batch vs send（send_batch 是项目写的包装层，send 是 Kafka 库提供的方法）
- send_batch 失败不抛异常（避免 SDK 误重试导致重复发送）
- Producer vs Consumer 对比（Producer 被动+不落库，Consumer 主动+真落库）
- Consumer 生命周期：start_consumer（创建+连接+create_task 启动后台循环）
- create_task（= Java new Thread + start，把死循环放后台协程）
- consume_loop 死循环：拉消息 → parse 解析 → 分到 5 桶 → 满了刷盘
- flush_map 策略模式（字典 + 函数引用代替 if-else，Python 优势）
- 两个刷盘条件：桶满 50 条 OR 5 秒超时
- finally 强制刷残留（关闭前把桶里没刷的数据强制写入，防丢失）
- 5 个 parse_* 数据解析（item → ClickHouse 表字段，含 calculate_cost 算钱）
- PARSE_MAP 路由（span_type → 对应 parse 函数，策略模式）
- 两层回退（item.get("x") or attrs.get("x", default)，先顶层后 attributes）
- 5 个 flush_* 批量写入（薄封装，真正重试逻辑在 _retry_insert）
- 完整 7 步数据流总结（SDK → Backend → Kafka → Consumer → ClickHouse）
- 程序分类：遥测系统（Telemetry System）跟前后端架构是并列的另一种程序类型

**关键认知**：
- Producer 是壳（send_batch），send 是芯（Kafka 库提供）
- await send() 等的是"丢进缓冲区"，不等 Kafka 确认
- 回调在 send_batch 返回后才执行，这就是"非阻塞"本质
- Consumer 才是真正落库的角色，Producer 只丢数据进 Kafka
- parse 在 append 之前执行，桶里装的是解析后的数据不是原始 item
- flush_map[key][1](batch) = 动态调用对应 flush 函数（索引 [1] 是第二个函数）
- finally 保证关闭前不丢数据
- 3 次重试失败丢弃数据是缺陷，生产环境要加死信队列

## ⚠️ 代码缺陷清单（学习过程中发现）

### 缺陷 1：ToolSDK 的 span_type 用错，Tool 数据没进 tool_calls 表

**位置**：`sdk/agent_insight_sdk/tool_sdk.py` 第 188 行 `_report_tool_span()`

**问题**：
- ToolSDK 上报 Tool 调用时 `span_type="trace"`，导致 Tool 数据只进了 `agent_traces` 表
- Tool 的具体内容（tool_name/input/output/duration）全塞在 `attributes` 字典里
- 后端明明有独立的 `tool_calls` 表（有专门字段），但没被使用

**对比 LLMInterceptor**（正确做法）：
- LLMInterceptor 一次 LLM 调用产生 3 条 SpanData（trace + llm_metrics + prompt），分别进 3 张表
- ToolSDK 一次 Tool 调用只产生 1 条 SpanData（只有 trace），应该产生 2 条（trace + tool_call）

**正确改法**：
```python
# 应该产生两条：
trace_span = SpanData(..., span_type="trace", attributes={"tool_name": tool_name})
tool_span = SpanData(..., span_type="tool_call", tool_name=..., input_data=..., output_data=..., duration_ms=..., status=...)
self._submit([trace_span, tool_span])
```

### 缺陷 2：client.py 的 insert_* 是 async 函数，但被 _retry_insert 用 run_in_executor 当同步函数跑

**位置**：`backend/app/clickhouse/client.py`

**调用链**：
```
consumer.py 第 267 行: await _retry_insert(insert_traces, batch, "traces")
                             ↓ 传的是 async 函数 insert_traces
client.py 第 46 行:    await loop.run_in_executor(None, insert_fn, data)
                             ↓ 把 async 函数丢进线程池当同步函数跑
```

**问题**：
- `insert_traces` 是 `async def`（第 64 行）
- `insert_traces` 内部已经自己用 `run_in_executor(None, _insert)` 把同步操作丢线程池了（第 100 行）
- 但 consumer.py 把 `insert_traces` 这个 async 函数传给 `_retry_insert`
- `_retry_insert` 又用 `run_in_executor(None, insert_fn, data)` 把它丢线程池
- **结果**：线程池里调用 async 函数只会返回 coroutine 对象，不会真正执行，导致静默失败

**两种修复方案**：
- 方案 A：`_retry_insert` 改成直接 `await insert_fn(data)`，不用 run_in_executor（因为 insert_* 内部已经处理了）
- 方案 B：`insert_*` 改成同步函数，只让 `_retry_insert` 做 run_in_executor（去掉 insert_* 里的 async + run_in_executor）

**薄弱点**：
- async/await 异步机制需要实战巩固（理解了但还没写过）
- Python 函数引用 + 字典代替策略模式的写法还不熟练
- ClickHouse 列存原理（为什么聚合快）还没深入
- SQL 注入风险已识别，参数化查询写法需要练习

### 2026-06-29 · clickhouse/client.py（ClickHouse 客户端）

**学了什么**：
- clickhouse-driver 是同步库，用 run_in_executor 丢线程池执行
- get_client 懒加载单例（第一次用到才创建，区别于 Producer/Consumer 启动时创建）
- _retry_insert 三机制：run_in_executor + 指数退避（1s/2s/4s）+ 3次失败丢弃
- 5 个 insert_* 套路：get_client → columns + values 列表推导式 → execute 参数化（安全）
- 6 个 query_* 套路：get_client → 拼 SQL → execute → dict(zip(columns, row)) 转字典
- dict(zip(columns, row)) 元组转字典（Java 要手动遍历 ResultSet 10 行，Python 一行）
- " OR ".join(...) 列表推导式拼接 SQL 条件
- 聚合查询是 OLAP 主场（group by + avg/sum，ClickHouse 秒杀）
- tool_stats 物化视图（预先聚合，查询比现算快）

**关键认知**：
- run_in_executor 期望同步函数，传 async 函数会静默失败（函数体不执行，系统以为成功）
- 静默失败比丢弃更危险（没日志，以为成功，数据其实没写）
- 修复方案：insert_* 改成同步函数（去掉 async）
- query_* 用 f-string 拼接 SQL 有 SQL 注入风险
- 修复方案：用参数化查询 `client.execute(query, {"trace_id": trace_id})`
- insert_* 用参数化（安全），query_* 用 f-string（危险）
- await 必须在事件循环里，run_in_executor 是线程池，井水不犯河水
- _retry_insert 被 consumer.py 的 flush_* 调用（不是本文件内部）
- 查询不用重试（失败返回 []），写入走重试（失败丢数据）

### 2026-06-30 · SDK agent_insight_sdk（完整 8 个文件）

**学了什么**：
- SDK 整体架构（门面模式 + 适配器模式 + 3 种代理模式）
- `__init__.py`：门面模式，统一导出 8 个 API，历史兼容（OpenAIInterceptor → LLMInterceptor）
- `context.py`：TraceContext + contextvars = Java ThreadLocal 增强版，协程安全，跨函数/协程传递 trace_id
- `tool_sdk.py`：装饰器自动埋点，@instrument = Java @AOP Around，同步/异步函数自动适配
- `trace_api.py`：手动埋点工具，with 上下文管理器自动清理，startTrace/startSpan/endSpan
- `uploader.py`：AsyncBatchUploader 异步批量上报，队列有界 + 背压保护 + 指数退避重试
- `stream_monitor.py`：流式响应监控，3 角色分工（Metrics 存数据 / Monitor 记时间 / Stream 包装迭代）
- `providers/base.py`：适配器基类 + LLMInterceptor 调度器 + LLMCallRecord 统一数据格式
- `providers/openai_compatible.py`：猴子补丁（运行时替换函数引用），流式响应两层包装
- 三种代理模式对比：装饰器（套自己函数）、猴子补丁（套别人函数）、包装器（套对象）
- LangChain 集成：补丁打在底层 openai client，上层 LangChain/LangGraph 全自动受益
- MCP 本质：远程 Tool，跟本地 Tool 没区别，用 @tool_sdk.instrument(tool_type="mcp") 包装

**关键认知**：
- 猴子补丁 = 运行时替换函数引用 = Java Spring AOP 动态代理，SDK 无侵入拦截 LLM 调用
- 装饰器 = 编译时/导入时织入 = Java AspectJ，适合套自己写的函数
- ContextVar = ThreadLocal 协程版，每个协程有独立副本，asyncio 场景安全
- 流式响应 TPS 定义：decode_ms ÷ output_tokens（越小越好，吞吐量越高）
- 导入即注册机制：import 模块时自动执行 register_adapter()，把 Adapter 加入全局列表
- Python 模块全局单例：整个程序 import 多少次都是同一个对象
- 依赖注入：TraceAPI/ToolSDK/LLMInterceptor 三大组件共享同一个 AsyncBatchUploader 实例
- 一次 LLM 调用产生 3 条 SpanData（trace + llm_metrics + prompt），分别进 3 张表
- 背压保护：生产者快消费者慢时，丢数据 + 告警，宁可丢监控不卡业务
- 门面模式好处：用户不用记具体文件路径，一行 `from agent_insight_sdk import *` 全搞定

**Python vs Java 对照**：
| Python | Java 等价 |
|--------|----------|
| `@dataclass` | Lombok `@Data` |
| `default_factory` | 构造函数初始化（每次 new 时执行） |
| `contextvars.ContextVar` | `ThreadLocal` + 协程支持 |
| `functools.wraps` | 动态代理保留原方法签名/注解 |
| `asyncio.Queue` | `ArrayBlockingQueue` |
| `asyncio.create_task` | `new Thread().start()` |
| 猴子补丁 | CGLIB 动态代理 |
| 装饰器 | AspectJ 编译时织入 |
| `__init__.py` 导出 | `Facade` 门面模式 |
