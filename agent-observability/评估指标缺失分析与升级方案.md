## 二、缺失指标全景（对照 12 指标框架）

### 2.1 检索指标（RAG 相关）— ❌ 全部缺失

| 指标 | 含义 | 测量方式 | 需要 ground truth？ | 学术来源 |
|------|------|---------|-------------------|---------|
| Context Relevance | 检索内容与问题的相关度 | LLM-as-Judge 打分 | ❌ | RAGAS |
| Context Recall | 需要的信息是否都检索到了 | LLM 逐句检查标准答案是否被覆盖 | ✅ 需要标准答案 | RAGAS |
| Context Precision | 检索结果中有效信息占比 | LLM 判断每个文档是否相关 + 排名 | ❌ | RAGAS |
| Retrieval Latency | 检索延迟 | 时间戳计算 | ❌ | 通用 |

### 2.2 生成指标 — ❌ 缺质量评分

| 指标 | 含义 | 测量方式 | 已有？ |
|------|------|---------|--------|
| Answer Faithfulness | 答案是否忠于证据（不编造） | LLM 逐句验证答案陈述是否可从证据推导 | ❌ 缺 |
| Answer Relevance | 答案是否回应了用户问题 | LLM 判断答案与问题的相关度 | ❌ 缺 |
| Hallucination Rate | 幻觉率 | 1 - Faithfulness | ❌ 缺 |

> prompt_logs 表有原始 prompt+response 数据，但**没有质量评分**。

### 2.3 Agent 行为指标 — ❌ 缺准确度评估

| 指标 | 含义 | 测量方式 | 已有？ |
|------|------|---------|--------|
| Tool Selection Accuracy | 工具选择准确度 | 对比 ground truth（应该调哪个工具） | ❌ 缺 |
| Tool Execution Success | 工具执行成功率 | status=success 占比 | ⚠️ 有 status 但没准确度 |
| Multi-Step Coherence | 多步连贯性 | LLM 判断步骤间逻辑是否连贯 | ❌ 缺 |
| 多 Agent 交接点评估 | 多 Agent 消息传递正确率 | LLM 检查交接点信息 | ❌ 缺 |

> tool_calls 表有 status(success/error)，但**没有"选对工具了吗"的评估**。

### 2.4 生产指标 — ⚠️ 部分缺失

| 指标 | 已有？ | 缺失 |
|------|--------|------|
| Cost per Query | ✅ cost_usd | - |
| P99 Latency | ❌ 只有单次 duration_ms | 缺 P50/P95/P99 分位计算 |
| N-run consistency | ❌ | 缺多次运行一致性 |
| Error type 分类 | ❌ 只有 status=error | 缺 error_type（timeout/connection/param_error/tool_not_found） |

### 2.5 轨迹质量 — ❌ 缺失

| 指标 | 含义 | 测量方式 |
|------|------|---------|
| Trajectory exact match | 是否走了最优路径 | 对比 golden trajectory |
| Trajectory precision | 实际步骤中必要的占比 | 实际步骤 ∩ 期望步骤 / 实际步骤 |
| Trajectory recall | 期望步骤是否都执行了 | 实际步骤 ∩ 期望步骤 / 期望步骤 |

## 三、RAG 评估指标测量方法（RAGAS 学术标准）

> 来源：RAGAS 论文（Retrieval Augmented Generation Assessment，2023 ES TriPOD 顶会）
> 工业实现：TruLens / DeepEval / LangSmith 都基于 RAGAS 框架

### 3.1 Context Relevance（检索内容相关度）

```python
def evaluate_context_relevance(question: str, retrieved_context: str,
                                judge_llm) -> float:
    """
    RAGAS Context Relevance
    原理：LLM 提取检索内容中的"有用句子"，除以总句子数
    """
    prompt = f"""
    用户问题：{question}
    检索到的内容：{retrieved_context}

    请判断：检索到的内容中，有多少句子对回答用户问题有帮助？
    返回 0-1 分数：1.0=全部相关，0.0=完全不相关
    """
    return float(judge_llm.chat(prompt))
```

### 3.2 Context Recall（信息覆盖度）

```python
def evaluate_context_recall(question: str, ground_truth_answer: str,
                             retrieved_context: str, judge_llm) -> float:
    """
    RAGAS Context Recall
    需要 ground truth（标准答案）
    原理：标准答案中的每个陈述，是否都能在检索内容中找到
    """
    prompt = f"""
    用户问题：{question}
    标准答案：{ground_truth_answer}
    检索到的内容：{retrieved_context}

    逐句检查标准答案：每个陈述是否能从检索内容推导？
    返回：能推导的陈述数 / 总陈述数
    """
    return float(judge_llm.chat(prompt))
```

### 3.3 Context Precision（有效信息占比）

```python
def evaluate_context_precision(question: str, retrieved_docs: list,
                                judge_llm) -> float:
    """
    RAGAS Context Precision
    原理：相关文档是否排在前面（类似 Precision@K）
    """
    scores = []
    for doc in retrieved_docs:
        prompt = f"问题：{question}\n文档：{doc}\n这个文档对回答有帮助吗？(1/0)"
        scores.append(int(judge_llm.chat(prompt)))

    # 加权：相关文档排在前面得分更高
    weighted = sum(s / (i + 1) for i, s in enumerate(scores) if s == 1)
    total_relevant = sum(scores)
    return weighted / total_relevant if total_relevant > 0 else 0
```

### 3.4 Answer Faithfulness（答案忠实度 = 幻觉检测）

```python
def evaluate_faithfulness(answer: str, retrieved_context: str,
                          judge_llm) -> float:
    """
    RAGAS Faithfulness
    原理：答案中的每个陈述，是否都能从证据中找到
    答案说了证据里没有的东西 = 不忠实 = 幻觉
    """
    # Step 1: 提取答案中的所有事实陈述
    statements = judge_llm.chat(f"提取以下回答中的所有事实陈述：\n{answer}")

    # Step 2: 逐条验证是否能从证据推导
    faithful = 0
    for stmt in statements:
        result = judge_llm.chat(f"陈述：{stmt}\n证据：{retrieved_context}\n能推导吗？(1/0)")
        faithful += int(result)

    return faithful / len(statements)
```

### 3.5 指标总结表

| 指标 | 测量方式 | 需要 ground truth？ | 输入 | 输出 |
|------|---------|-------------------|------|------|
| Context Relevance | LLM 打分 | ❌ | question + retrieved_context | 0-1 分数 |
| Context Recall | LLM 逐句检查 | ✅ 需要标准答案 | question + ground_truth + retrieved_context | 0-1 分数 |
| Context Precision | LLM 逐文档判断 + 排名 | ❌ | question + retrieved_docs 列表 | 0-1 分数 |
| Answer Faithfulness | LLM 提取陈述 + 逐条验证 | ❌ | answer + retrieved_context | 0-1 分数 |
| Answer Relevance | LLM 打分 | ❌ | question + answer | 0-1 分数 |

## 四、LLM-as-Judge 可靠性问题

### 4.1 已知偏差

| 偏差类型 | 表现 | 解法 |
|---------|------|------|
| 冗长偏好 (verbose bias) | 偏好长的答案 | rubric 明确"简洁性"标准 |
| 位置偏好 (position bias) | 受选项呈现顺序影响 | 随机打乱顺序 + 多次取平均 |
| 时间漂移 (drift) | 评分标准随时间变化 | 定期用人工标注样本校准 |
| 幻觉 | 裁判 LLM 自己幻觉 | 多次评判取共识 + 人工抽检 |

### 4.2 工业实践：三层评判

```
第一层：确定性规则（正则/Schema）→ 快、免费、无幻觉
    ↓ 过滤不了的
第二层：小 LLM 判断 → 便宜、快、但有噪声
    ↓ 关键场景
第三层：人工抽检 → 慢、贵、但最准
```

### 4.3 LLM-as-Judge 五条硬规则

1. 用**结构化 rubric** 约束评判标准
2. **多次评判取平均**减少方差（跑 3 次取均值）
3. 用**人工标注样本校准**（检测系统性偏差）
4. 将复杂评估**分解为简单检查**
5. **监控评判一致性**防止漂移

## 五、评估工具底层机制对比

### 5.1 工具对比

| 工具 | 类型 | 底层机制 | 开源 | 适合 |
|------|------|---------|------|------|
| LangSmith | 商业 | Trace + LLM-as-Judge + Dataset 管理 | 否 | LangChain 团队 |
| Langfuse | 开源 | OTel 标准 + Trace + 在线评估 | MIT | 自托管 |
| Arize Phoenix | 开源 | OTel 原生 + RAG 评估最佳 | Elastic 2.0 | RAG 系统 |
| DeepEval | 开源 | pytest 风格 + 50+ 指标 + CI 集成 | Apache 2.0 | 开发者 CI |

### 5.2 底层机制三问

**Q1：底层评估机制是什么？**
- 不只是获取数据，还会对数据**进行处理和评估**
- 核心是**评估器（Evaluator）**：对每条 Trace 运行规则检查或 LLM-as-Judge 打分

**Q2：指标阈值如何确定？**

| 方式 | 说明 | 例子 |
|------|------|------|
| 行业基准 | 业界共识 | Task completion < 85% 告警 |
| 历史数据 | 过去 N 天统计 | P99 > 历史 P99 × 1.5 告警 |
| 业务定义 | 根据需求定 | 成本 > 日预算告警 |

**Q3：评判方式是什么？**
- 不是只用小模型，是**确定性规则 + LLM-as-Judge + 人工**三层结合
- 确定性规则：正则/Schema/代码检查（快、免费、无幻觉）
- LLM-as-Judge：用 LLM 打分（能检查语义，但有偏差）
- 人工：关键场景抽检（最准但慢贵）

## 六、项目升级建议（三阶段）

### Phase 1：补质量评分（最小改动）

在 `llm_metrics` 表加列：

```sql
ALTER TABLE llm_metrics ADD COLUMN faithfulness_score Float64 DEFAULT 0;
ALTER TABLE llm_metrics ADD COLUMN relevance_score Float64 DEFAULT 0;
ALTER TABLE llm_metrics ADD COLUMN hallucination_rate Float64 DEFAULT 0;  -- 1 - faithfulness
```

在 SDK 中增加 LLM-as-Judge 调用：

```python
# 在 LLM 调用结束后，异步调用 Judge LLM 打分
async def evaluate_quality(response: str, context: str, question: str):
    faithfulness = await judge_llm.evaluate_faithfulness(response, context)
    relevance = await judge_llm.evaluate_relevance(question, response)
    return {"faithfulness": faithfulness, "relevance": relevance}
```

### Phase 2：补 RAG 指标（如果有 RAG）

新增 `rag_metrics` 表：

```sql
CREATE TABLE IF NOT EXISTS rag_metrics (
    trace_id String,
    span_id String,
    question String,
    retrieved_context String,
    ground_truth String DEFAULT '',        -- 可选：标准答案
    context_relevance Float64 DEFAULT 0,   -- LLM-as-Judge
    context_recall Float64 DEFAULT 0,      -- 需要 ground_truth
    context_precision Float64 DEFAULT 0,   -- LLM-as-Judge
    answer_faithfulness Float64 DEFAULT 0, -- LLM-as-Judge
    answer_relevance Float64 DEFAULT 0,    -- LLM-as-Judge
    retrieval_latency_ms Float64 DEFAULT 0,
    created_at DateTime DEFAULT now()
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(created_at)
ORDER BY (trace_id, created_at);
```

### Phase 3：补 Agent 行为指标

在 `tool_calls` 表加列：

```sql
ALTER TABLE tool_calls ADD COLUMN tool_selection_correct UInt8 DEFAULT 0;  -- 选对了吗
ALTER TABLE tool_calls ADD COLUMN args_valid UInt8 DEFAULT 0;              -- 参数格式对吗
ALTER TABLE tool_calls ADD COLUMN error_type String DEFAULT '';            -- 错误分类
```

错误分类枚举：

```python
ERROR_TYPES = {
    "timeout": "工具执行超时",
    "connection": "网络连接错误",
    "param_error": "参数格式错误",
    "tool_not_found": "工具不存在（幻觉）",
    "auth_error": "权限不足",
    "rate_limit": "限流",
    "unknown": "未知错误"
}
```

新增 `trajectory_metrics` 表（轨迹质量评估）：

```sql
CREATE TABLE IF NOT EXISTS trajectory_metrics (
    trace_id String,
    actual_steps UInt32,              -- 实际步数
    expected_steps UInt32 DEFAULT 0,  -- 期望步数（golden trajectory）
    exact_match UInt8 DEFAULT 0,      -- 是否完全匹配最优路径
    precision Float64 DEFAULT 0,      -- 必要步骤占比
    recall Float64 DEFAULT 0,         -- 期望步骤覆盖率
    n_run_consistency Float64 DEFAULT 0, -- N次运行一致性
    created_at DateTime DEFAULT now()
) ENGINE = MergeTree()
ORDER BY (trace_id, created_at);
```

## 八、评估管线三阶段

```
Phase 1：离线回归（上线前）
    ├── Golden Dataset（100-500 样本）
    ├── CI 每次 PR 跑快照
    ├── 阈值门禁（faithfulness ≥ 0.85 / p95 < 2s）
    └── 工具：DeepEval + pytest

Phase 2：在线/影子评估（灰度期）
    ├── 生产流量采样 10-20%
    ├── LLM-as-Judge 实时打分
    ├── A/B 测试新版本
    └── 工具：LangSmith / Langfuse

Phase 3：持续监控（稳定期）
    ├── 全链路实时监控
    ├── 自动告警
    ├── 失败案例自动收集 → 回馈 Golden Dataset
    └── 工具：Langfuse / Phoenix / 本项目
```
