### 2.2 核心数据流（7 步全链路）

```
用户 Agent 代码执行
    │
    ├─ LLM 调用 → Monkey Patch 拦截 → 生成 3 条 SpanData
    │                                (trace + llm_metrics + prompt)
    │
    ├─ Tool 调用 → 装饰器拦截 → 生成 1 条 SpanData
    │                           (tool_call) ← 已修复
    │
    └─ SpanData → AsyncBatchUploader 队列
                     │
                     ├─ 满 20 条或 0.5s → 批量 HTTP POST
                     │
                     ▼
              Backend collect.py (校验 → 202)
                     │
                     ▼
              Kafka (削峰)
                     │
                     ▼
              Consumer (parse → 分桶 → flush)
                     │
                     ▼
              ClickHouse 5 张表
                     │
                     ▼
              Query API → 前端 Dashboard
```

### 2.3 三种拦截方式对比

| 方式 | 适用场景 | 对应 Java | 代码位置 |
|------|---------|-----------|---------|
| **Monkey Patch** | 拦截第三方 LLM SDK | Spring AOP 动态代理 (CGLIB) | `openai_compatible.py` |
| **装饰器 @instrument** | 拦截自己写的 Tool 函数 | AspectJ 编译时织入 | `tool_sdk.py` |
| **包装器 MonitoredStream** | 拦截流式迭代器对象 | 包装器模式 (Wrapper Pattern) | `stream_monitor.py` |

### 2.4 5 种 span_type 与 5 张表对应关系

| span_type | 写入的表 | 谁产生的 | 关键字段 |
|-----------|---------|---------|---------|
| `trace` | agent_traces | LLM + Tool 都产生 | trace_id, span_id, parent_span_id, name |
| `llm_metrics` | llm_metrics | LLM 调用 | model, prefill_ms, decode_ms, tps, tokens |
| `prompt` | prompt_logs | LLM 调用 | prompt, response, input_tokens, output_tokens |
| `tool_call` | tool_calls | Tool 调用 | tool_name, input_data, output_data, duration_ms |
| `session` | sessions | TraceAPI | session_id, agent_name, trace_count |

## 三、易错点清单（学习过程中的理解错误）

### ❌ 易错点 1：以为 Tool 数据进了 tool_calls 表

| 维度 | 我的错误理解 | 实际情况（修复前） | 实际情况（修复后） |
|------|------------|-------------------|-------------------|
| span_type | Tool 数据进 tool_calls 表 | ❌ span_type="trace"，只进 agent_traces | ✅ span_type="tool_call"，正确进 tool_calls |
| 字段存储 | 结构化存储 | ❌ 全塞在 attributes JSON 里 | ✅ 结构化字段 |
| 查询方式 | 从 tool_calls 表查 | ❌ API 也只查 agent_traces | ✅ 可从 tool_calls 查 |

**教训**：看代码不能只看表设计，要看实际写入的 span_type 值。

---

### ❌ 易错点 2：以为 LLM 返回的 tool_calls 决策被记录了

| 维度 | 我的错误理解 | 实际情况 |
|------|------------|---------|
| LLM 决策记录 | LLM 返回的 tool_calls JSON 被提取 | ❌ `_extract_response()` 只提取 `message.content`，完全无视 `message.tool_calls` |
| 父子关联 | LLM → Tool 有直接关联 | ⚠️ 只有通过 ContextVar 的时间顺序父子关系，没有 tool_call_id 直接关联 |
| 验证一致性 | 能验证 LLM 决策和实际执行一致 | ❌ 无法验证 |

**教训**：观测系统记录的是"执行了什么"，不是"LLM 想执行什么"。

---

### ❌ 易错点 3：以为 async 函数传给 run_in_executor 能正常执行

| 维度 | 我的错误理解 | 实际情况 |
|------|------------|---------|
| run_in_executor | 能执行任何函数 | ❌ 只能执行同步函数，传 async 函数只返回 coroutine 对象不执行 |
| 双重包装 | 无所谓 | ❌ insert_* 内部已 run_in_executor，外面再包一层导致静默失败 |
| 静默失败 | 会报错 | ❌ 没有异常抛出，系统以为成功，数据静默丢失 |

**教训**：async 函数不能丢进线程池，await 必须在事件循环里。

---

### ❌ 易错点 4：把内部实现细节和用户使用方式混在一起

| 维度 | 我的错误理解 | 实际情况 |
|------|------------|---------|
| 用户操作 | 用户要管内部包装几次 | ✅ 用户只需 `interceptor.wrap(client)` 一次 |
| 内部细节 | 不用关心 | SDK 内部怎么包装是 SDK 自己的事 |

**教训**：区分 SDK 用户接口和 SDK 内部实现。

---

### ❌ 易错点 5：混淆装饰器和 Java 注解

| 维度 | 装饰器 | Java 注解 |
|------|--------|----------|
| 执行时机 | 运行时自己执行 | 需要框架扫描 |
| 本质 | 接收函数返回新函数 | 标记标签 |
| 类比 | 更像 AOP 环绕通知 | 更像配置标记 |

**教训**：装饰器是"自己干活的函数"，Java 注解是"需要框架扫描的标签"。

## 四、业界 LLM 可观测性方案对比

### 4.1 业界主流方案概览

| 方案 | 类型 | 后端存储 | SDK 拦截方式 | 开源 |
|------|------|---------|-------------|------|
| **Langfuse** | 开源平台 | ClickHouse + Postgres + Redis + S3 | OpenTelemetry + Monkey Patch | ✅ MIT |
| **LangSmith** | 商业 SaaS | SmithDB (Rust) | LangChain Callback + OTel | ❌ 闭源 |
| **OpenTelemetry GenAI** | 标准 | 任意 (OTLP 协议) | 语义约定 + Instrumentation 库 | ✅ CNCF |
| **Datadog LLM Obs** | 商业 APM | Datadog 自有 | ddtrace 装饰器 + Monkey Patch | ❌ 闭源 |
| **Braintrust** | 商业平台 | 自有 | 框架适配器 + OTel | ❌ 闭源 |
| **本项目 Agent-Insight** | 学习项目 | ClickHouse + Kafka | Monkey Patch + 装饰器 + 包装器 | ✅ 学习用 |

### 4.2 Langfuse 架构（最接近的业界方案）

```
SDK (Python/JS, 基于OTel)
    │
    ▼
API (Next.js Web 容器)
    │
    ├──→ S3/Blob (存原始事件)
    │
    ▼
Redis Queue (BullMQ, 解耦)
    │
    ▼
Worker (Express, 异步处理)
    │
    ├──→ ClickHouse (traces/observations/scores)
    └──→ Postgres (用户/项目/API Key/Prompt管理)
```

**Langfuse vs 本项目对比：**

| 维度 | Langfuse | 本项目 | 差距 |
|------|---------|--------|------|
| **存储架构** | ClickHouse + Postgres + Redis + S3 | ClickHouse + Kafka | 缺少 S3 存大文件、缺少 Postgres 存元数据 |
| **异步处理** | Redis Queue + Worker 独立部署 | Kafka + Consumer | 方案类似，Kafka 比 Redis Queue 更重但更可靠 |
| **SDK 拦截** | OpenTelemetry 标准 + Monkey Patch | 自研 Monkey Patch + 装饰器 | 缺少 OTel 标准，但有自研适配器模式 |
| **批量上报** | SDK 端 batch + Worker 端 batch | SDK 端 batch + Consumer 端 batch | ✅ 两级批量一致 |
| **背压保护** | S3 存储 + Redis 队列解耦 | asyncio.Queue 有界 + 丢数据 | 都有背压，策略不同 |
| **重试机制** | ClickhouseWriter 指数退避 + 批量拆分 | 指数退避 3 次 | Langfuse 更完善（批量拆分+截断） |
| **Prompt 管理** | 版本控制 + Playground | 无 | 缺少 Prompt 管理能力 |
| **评估能力** | LLM-as-Judge + 人工标注 + 代码评估 | 无 | 缺少质量评估 |
| **多模态** | S3 存图片/音频 | 无 | 缺少多模态支持 |

### 4.3 OpenTelemetry GenAI 语义约定（业界标准）

OTel GenAI SIG 定义了 6 层语义约定，本项目可以借鉴：

| OTel 层级 | 定义 | 本项目对应 | 差距 |
|-----------|------|-----------|------|
| **Layer 1: Client Spans** | LLM 调用标准化 (model/tokens/latency) | ✅ llm_metrics 表 | 基本对齐 |
| **Layer 2: Agent Spans** | Agent 生命周期 (create_agent/invoke_agent) | ⚠️ trace 表只记轨迹 | 缺少 Agent 级别 span |
| **Layer 3: MCP Spans** | MCP 工具调用标准化 | ❌ 无 | 未实现 |
| **Layer 4: Events** | 内容捕获 (prompt/response/tool_call) | ✅ prompt_logs 表 | 基本对齐 |
| **Layer 5: Metrics** | 两个核心 Histogram (duration/token) | ✅ llm_metrics 表 | 基本对齐 |
| **Layer 6: Provider** | 厂商特定属性 | ✅ 适配器模式 | 基本对齐 |

**OTel 的 tool_call 标准属性（本项目可以借鉴）：**
```
gen_ai.tool.call.id       → 工具调用唯一 ID（关联 LLM 决策和执行）
gen_ai.tool.call.arguments → LLM 生成的参数（不是实际执行的参数）
gen_ai.tool.call.result    → 工具执行结果
gen_ai.tool.name           → 工具名称
gen_ai.tool.type           → 工具类型 (function/extension/datastore)
```

### 4.4 可借鉴的业界实践

| 借鉴点 | 来源 | 怎么借鉴 |
|--------|------|---------|
| **tool_call_id 关联** | OTel GenAI | 在 LLM 的 attributes 里加 tool_call_id，Tool 的 span 里也加，实现双向关联 |
| **LLM 决策记录** | OTel GenAI | 在 `_extract_response()` 里提取 `message.tool_calls` JSON |
| **Fail-Open 设计** | Agent Observatory | 观测层失败永远不影响 Agent 执行（本项目已有背压丢数据，可进一步明确） |

| **LLM-as-Judge 评估** | Langfuse/LangSmith | 加自动评估层，对 LLM 输出打分 |
| **批量拆分重试** | Langfuse ClickhouseWriter | 写入失败时把批次拆成两半，一半重试一半重新排队 |

| **计算-存储分离** | Langfuse | ClickHouse 读写分离，查询走只读节点 |

## 五、与老师交流的技术问题清单

### 🔥 高优先级问题（直接关系到项目质量）

#### Q1: Tool Call 关联问题
> "老师，我发现 LLM 返回的 `tool_calls` 决策信息（工具名、参数、tool_call_id）目前没有被提取和记录。这导致我们只能从时间顺序推测 LLM 调用了哪个工具，但没法验证 LLM 决策和实际执行是否一致。OpenTelemetry GenAI 规范里定义了 `gen_ai.tool.call.id` 来做双向关联，咱们后续有没有计划补上这个？"

#### Q2: Agent 级别 Span 缺失
> "OTel GenAI 规范定义了 `create_agent` 和 `invoke_agent` 两种 Agent 生命周期 Span，咱们目前只有 trace/llm_metrics/prompt/tool_call/session 五种 span_type，缺少 Agent 级别的 Span。如果一个系统里有多个 Agent 协作（Multi-Agent），现在的架构能区分是哪个 Agent 发起的调用吗？"

#### Q3: 评估能力空白
> "Langfuse 和 LangSmith 都有 LLM-as-Judge 自动评估能力，能对 LLM 输出质量打分。咱们目前只采集了执行数据（耗时、token、输入输出），但缺少质量评估。后续有没有计划加评估层？"

#### Q4: OpenTelemetry 标准兼容
> "业界趋势是全面拥抱 OpenTelemetry，Langfuse v3 已经基于 OTel 重建 SDK。咱们的 SDK 目前是自研的 SpanData 格式，后续有没有计划兼容 OTLP 协议？这样就能对接 Datadog、Grafana、Jaeger 等任意后端了。"

### 🤔 中优先级问题（架构设计思考）

#### Q5: Kafka vs Redis Queue
> "Langfuse 用 Redis (BullMQ) 做消息队列解耦，咱们用 Kafka。Kafka 更重但更可靠，Redis 更轻但可能丢数据。老师选 Kafka 的主要考虑是什么？是看重持久化能力还是削峰性能？"

#### Q6: 存储分层设计
> "Langfuse 用 Postgres 存元数据（用户/API Key/Prompt 版本），ClickHouse 只存 tracing 数据。咱们目前所有数据都在 ClickHouse，后续如果加用户体系和 Prompt 管理，是不是也要做存储分层？"

#### Q7: 背压策略
> "咱们目前的背压策略是队列满时丢数据 + 告警。Langfuse 的做法是先写 S3 再异步处理，不会丢数据。这两种策略的取舍是什么？咱们是考虑到实现复杂度还是觉得丢监控数据可接受？"

#### Q8: ClickHouse 写入失败处理
> "Langfuse 的 ClickhouseWriter 在写入失败时会做批量拆分（把批次分成两半，一半重试一半重新排队），还有单字段截断保护（超 1MB 截断）。咱们的重试只是指数退避 3 次后丢弃，要不要借鉴这种更精细的失败处理？"

### 💡 探索性问题（技术深度）

#### Q9: Monkey Patch 的局限性
> "Monkey Patch 能拦截 OpenAI SDK 的 `client.chat.completions.create`，但如果用户直接用 httpx/requests 调用 OpenAI API（不走 SDK），就拦截不到。Langfuse 通过 OTel 的 HTTP Instrumentation 能拦截任意 HTTP 调用。咱们要不要加一层 HTTP 级别的拦截？"

#### Q10: Multi-Agent 场景
> "如果未来要支持 Multi-Agent 协作（Agent A 调用 Agent B），现有的 ContextVar 父子关系能正确传递吗？会不会出现上下文串台的问题？OTel 的分布式追踪是通过 trace context header 跨进程传递的，咱们要不要考虑这个？"

#### Q11: 敏感数据脱敏
> "prompt_logs 表存了完整的 Prompt 和 Response，可能包含用户敏感信息。Langfuse 有 PII 脱敏能力。咱们后续要不要加脱敏层？在 SDK 端脱敏还是 Consumer 端脱敏？"

#### Q12: 成本计算精度
> "Consumer 里的 `calculate_cost` 是按模型名查价格表算的，但不同厂商的定价模型不一样（有的按 input/output 分别计价，有的有缓存折扣）。Langfuse 和 LangSmith 都支持更精细的成本计算。咱们的成本计算精度够用吗？"

## 六、学习总结与自我评估

### 6.1 已掌握的核心概念

| 概念 | 掌握程度 | 说明 |
|------|---------|------|
| Monkey Patch 原理 | ✅ 理解 | 运行时替换函数引用 = Java CGLIB 动态代理 |
| 装饰器原理 | ✅ 理解 | 接收函数返回新函数 = Java AOP 环绕通知 |
| ContextVar | ✅ 理解 | = Java ThreadLocal 协程版 |
| Trace/Span 模型 | ✅ 理解 | trace_id + span_id + parent_span_id 构建调用树 |
| 异步批量上报 | ✅ 理解 | 队列 + 批量 + 背压 + 重试 |
| 适配器模式 | ✅ 理解 | BaseProviderAdapter + 多厂商支持 |
| 流式响应监控 | ✅ 理解 | prefill_ms / decode_ms / tps |
| Kafka 削峰 | ✅ 理解 | Producer 非阻塞投递 + Consumer 批量消费 |
| ClickHouse 列存 | ✅ 理解 | OLAP 聚合快，适合监控数据 |
