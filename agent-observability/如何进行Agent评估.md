# 如何进行 Agent 评估

> 📅 创建：2026-07-16
> 🎯 目标：理解 LLM 评估与 Agent 评估的区别，掌握工业上常用的 Agent 评估方法

---

## 一、LLM 评估 vs Agent 评估

### 核心区别：评估「模型」还是评估「系统」

| 维度 | LLM 评估 | Agent 评估 |
|------|---------|-----------|
| **评估对象** | 单个模型 | 整个系统（模型 + 工具 + 记忆 + 编排） |
| **交互模式** | 单轮/少轮文本对话 | 多轮环境交互 |
| **能力范围** | 语言理解与生成 | 规划、推理、工具使用、环境感知 |
| **评估指标** | 困惑度(Perplexity)、BLEU、ROUGE、人工评分 | 任务完成率、工具选择准确率、执行效率、鲁棒性 |
| **输出确定性** | 相同输入基本相同输出 | 相同输入可能走完全不同的执行路径 |
| **失败模式** | 输出文本质量差 | 工具调错、参数错误、死循环、状态丢失 |
| **类比** | 考试答题（评分标准答案） | 软件系统测试（测端到端行为） |

### 为什么传统 LLM 评估不适用于 Agent？

1. **非确定性执行路径**：同一个输入，Agent 可能走完全不同的工具调用序列、推理步骤。传统 golden dataset 评估假设评估面是稳定的，对 Agent 不成立。

2. **误差累积效应**：12 步的 Agent 工作流中，第 2 步的小错误会污染后续所有步骤的上下文。到第 8 步时，Agent 在基于 6 步前的错误认知做推理。传统逐轮评分检测不到这种级联失败。一个 20 步工作流，每步 95% 可靠性，整体成功率只有 36%。

3. **工具调用维度**：Agent 可能调错 API、传错参数、误读工具返回值。最终输出看起来合理，但基于一个静默的工具失败。只评估最终文本会完全遗漏这种失败。

4. **跨会话状态管理**：多轮 Agent 携带状态（记忆、用户偏好、约束条件）。状态腐化（Agent 忘记用户在第 3 轮设定的约束）对单轮评估框架不可见。

### 一句话总结

> **评估 LLM 是评分单个回答；评估 Agent 是评分完整轨迹——工具调用序列、检索结果、中间状态、副作用。** 答案可能"对的原因错"或"错的因为工具给了脏数据"。

## 二、工业上常用的 Agent 评估方法

### 方法 1：任务完成率评估（Task Success Rate）

**思路**：Agent 是否成功完成了用户指定的任务？

**实现方式**：
- 准备一组测试用例（输入 + 期望结果）
- 运行 Agent，判断最终结果是否达标
- 统计成功率

**适用场景**：有明确成功标准的任务（如"帮我订一张明天北京的机票"）

**局限**：只看最终结果，不关心中间过程。答案对了但过程错了（走了弯路）也算成功。

### 方法 2：轨迹评估（Trajectory Evaluation）

**思路**：不只看最终答案，评估 Agent 的完整执行轨迹——工具调用序列、调用顺序、中间推理。

**核心概念——Golden Trajectories（黄金轨迹）**：
- 选取 5-10 条关键路径
- 对整条运行轨迹做断言：调了哪些工具、什么顺序、检索了什么上下文、最终状态
- "通过错误工具调用得到的正确答案"是一个伪装成绿勾的潜在事故

**适用场景**：对过程正确性要求高的场景（如金融交易、医疗诊断）

**局限**：需要更复杂的评估基础设施，非确定性路径难以精确匹配。

### 方法 3：LLM-as-Judge（LLM 充当评判员）

**思路**：用一个强 LLM（如 GPT-4）充当评判员，对 Agent 的输出打分。

**实现方式**：
- 给评判 LLM 一个评分标准（rubric）
- 输入 Agent 的输出和期望结果
- 评判 LLM 返回分数和理由

**适用场景**：开放性任务，没有唯一正确答案（如"帮我写一封邮件"）

**四种失败模式**：
1. **位置偏见**：倾向于选择第一个或最后一个选项
2. **冗长偏见**：倾向于给更长的回答更高分
3. **自评偏见**：倾向于给自己的输出更高分
4. **格式敏感**：评分受格式影响而非内容质量

**适用边界**：适合离线小样本评分，不适合生产环境逐轮实时评分（成本高、方差大）。

### 方法 4：Agent-as-a-Judge（Agent 充当评判员）

**思路**：LLM-as-Judge 的升级版——用 Agent 评估 Agent。评判者本身也是一个 Agent，具备工具使用、记忆、多步推理能力。

**与 LLM-as-Judge 的区别**：
- LLM-as-Judge：只看最终输出，给一个分数
- Agent-as-a-Judge：能观察被评估 Agent 的完整执行过程，对每一步给出反馈

**适用场景**：需要细粒度评估中间步骤的场景（如代码助手、Web 导航 Agent）

### 方法 5：工具契约测试（Tool Contract Testing）

**思路**：把每个工具/MCP 端点当作外部依赖，测试其契约：预期 schema、延迟、失败模式。

**实现方式**：
- 注入脏数据、畸形响应、超时
- 确认 Agent 能安全降级，而不是自信地传递错误值

**适用场景**：生产环境，确保 Agent 对工具故障有容错能力

### 方法 6：混合评估（Hybrid Evaluation）

**思路**：自动评分 + 人工评判结合，取两者优势。

**自动评分**：LLM-as-Judge、轨迹分析、负载测试 -> 可重复、可规模化
**人工评判**：语气、信任度、上下文适切性 -> 捕捉自动评分遗漏的细微质量

**工业最佳实践**：混合评估是必须的，不是可选的。

## 三、Agent 评估的核心维度

| 维度 | 评估内容 | 示例指标 |
|------|---------|---------|
| **任务完成率** | 是否成功完成用户任务 | 成功/失败、完成率% |
| **工具选择准确率** | 是否选了正确的工具 | 选对工具的比例 |
| **参数提取准确率** | 传给工具的参数是否正确 | 参数匹配率 |
| **执行效率** | 完成任务用了多少步/Token | 步数、Token 数、成本 |
| **延迟** | 从提问到回答多久 | 端到端延迟、首 token 耗时 |
| **鲁棒性** | 异常输入/工具失败时的处理 | 错误恢复率 |
| **安全性** | 是否执行了危险操作 | 危险操作次数 |
| **一致性** | 多次运行结果是否一致 | 方差、一致率 |

## 四、主流 Agent Benchmark

| Benchmark | 评估方向 | 特点 |
|-----------|---------|------|
| **AgentBench** | 8 个真实环境综合评测 | 覆盖操作系统、数据库、Web 等 |
| **SWE-bench** | 软件工程任务 | 修 GitHub Issue，测代码能力 |
| **WebArena** | Web 操作 | 浏览器自动化任务 |
| **GAIA** | 通用 AI 助手 | 多步推理 + 工具使用 |
| **τ-Bench (TAU-bench)** | 工具使用 + 策略遵循 | 企业级场景（航空/零售客服） |

## 六、生产环境评估最佳实践

### 1. 可重放的溯源（Replayable Provenance）
记录每一步的 request -> tool -> response -> outcome，可以确定性回放。这是"Agent 失败了"和"CRM 服务器在 14:03 返回了两天前的旧记录，Agent 信任了它"的区别。

### 2. 持续评估管线
- 离线：golden trajectory + LLM-as-Judge 小样本评分
- 在线：生产环境可观测性（trace 分析、错误聚类）
- 闭环：生产失败 -> 自动标注 -> 生成评估用例 -> 回归测试

### 3. 失败模式聚类
自动将相似失败模式分组——"15% 的会话出现了这种特定类型的工具调用循环"而不是"这里有 847 条失败的 trace"。

### 4. 五大生产评估维度
1. **行为胜过基准**：任务成功率、错误恢复、真实变异性下的一致性
2. **混合评估**：自动指标提供规模化可重复性，人工判断捕捉信任和可用性
3. **运营约束**：延迟、成本、工具可靠性、策略合规
4. **安全治理**：红队测试、PII 处理、权限边界
5. **用户体验**：信任度、上下文适切性

## 七、与本项目（Agent-Insight）的关联

本项目的可观测性系统为 Agent 评估提供**数据基础**：

| 评估维度 | 本项目提供的数据 | 对应表 |
|---------|----------------|--------|
| 任务完成率 | trace 链路 + session 状态 | agent_traces + sessions |
| 工具选择准确率 | tool_call 记录（工具名、参数、结果） | tool_calls |
| 执行效率 | llm_metrics（Token 数、耗时） | llm_metrics |
| 延迟 | prefill_ms / decode_ms / latency_ms | llm_metrics + prompt_logs |
| 鲁棒性 | error 字段 + status 字段 | 所有表 |
| 成本 | cost_usd（自动计算） | llm_metrics |

**可观测性是评估的前提**：没有 trace 数据，就无法做轨迹评估；没有 metrics 数据，就无法量化效率；没有 prompt_logs，就无法回放分析。
