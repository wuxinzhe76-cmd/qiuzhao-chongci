## 维度一：技术选型与整体架构（3题）

### Q1. 你的项目同时用了 PostgreSQL、Redshift、DynamoDB 三种存储，为什么不统一用一种？

**满分回答：**

这是一个"用合适的工具解决合适的问题"的典型架构决策，三者职责完全不同：

- **PostgreSQL（主库，JPA/Hibernate）**：存储业务运营数据——任务执行记录（Job）、投放报告（ImpressionReport）、资产配置（Asset）等。这类数据有强事务需求，字段结构稳定，需要外键约束和 Flyway 版本化迁移，JPA 的对象映射在这里开发效率最高。

- **AWS Redshift（分析库，MyBatis）**：存储每小时粒度的曝光量、流量、触达率等海量时序数据。Redshift 是列式存储，专为大规模聚合查询优化，可以在秒级完成对上亿行数据的 SUM/AVG 分析。这里选 MyBatis 而非 JPA，是因为查询语句需要**动态构造列名**（不同 population 段对应不同列名，如 `traffic`、`traffic_departure`、`traffic_population_3`），JPA 的 JPQL 不支持动态列名，MyBatis 的 `${}` 字符串替换配合 `StatementType.STATEMENT` 是唯一可行方案。

- **DynamoDB（KV 存储，Spring Data DynamoDB）**：存储网络拓扑（Network）、广告牌配置（Asset）、系数表（Coefficient）等相对静态的配置数据。DynamoDB 的 Key-Value 模型非常适合按 `platform-env-assetCode` 快速点查，读延迟稳定在个位数毫秒，且无需维护表结构。

**关键 Trade-off**：三套存储引入了多数据源管理复杂度，项目通过 `PostgresDsConfig`、`RedshiftConfig`、`DynamoDBConfig` 三个独立配置类隔离各自的连接池和事务管理，保证互不干扰。如果强制统一用 PostgreSQL，分析查询性能会下降 10-100 倍；统一用 Redshift 则无法支持复杂事务和 ORM；统一用 DynamoDB 则无法做复杂的多表 JOIN 报告。

### Q2. 同一套代码服务 AAM/MAM/SAM 三个平台，为什么不拆成三套独立服务？

**满分回答：**

这是"单代码库 Multi-tenant"模式 vs "多代码库 Multi-service"模式的架构取舍。

**选择单代码库的核心理由**：
1. **三个平台的业务逻辑 90% 相同**——曝光量计算、日期映射、报告生成的流程完全一致，只有 Population 数量（AAM=1、MAM=2、SAM=3）、SOT 允许值、S3 存储桶、Redshift Schema 等参数不同。
2. **Bug 修复只需一次**——如果修复 `setAvailableDates` 的日期映射 Bug，三个平台同时生效，不存在三套代码各自维护的版本漂移问题。
3. **CI/CD 简单**——`.gitlab-ci.yml` 通过分支名（如 `mam-master`）解析出 platform 和 environment，打包时注入 `-Dapp.platform=mam`，一套流水线覆盖三平台六环境。

**Spring Profile 差异化配置的实现**：
`application-mam.properties` 中设置 `dc.generic.am.population={"1":"traffic","2":"traffic_departure"}`，注入到 `ImpressionService` 的 `LinkedHashMap<Integer, String> populations` 字段。SQL 生成时，`getImpressionsSumParam` 遍历这个 Map，为每个 population key 生成对应的 `SUM(traffic_xxx)` 片段——这就是平台差异从配置一路流向 SQL 的完整链路。

**什么时候该拆**：如果三个平台出现**大量差异化业务逻辑**（比如 SAM 需要接入完全不同的计费系统），单代码库的 `if (platform.equals("sam"))` 条件判断会越来越多，维护成本超过合并收益，这时候才应该拆分——符合"过度设计比不足设计更危险"的工程原则。

### Q3. 投前预估和投后复盘在数据模型和一致性上有什么区别？

**满分回答：**

这是系统中最核心的业务区别，直接决定了两条不同的代码路径：

**数据来源不同**：
- **投前（Pre-campaign）**：用户选择未来的投放日期，系统没有这段时间的真实数据。`ImpressionReportService.generateReport` 会通过 `getRefDate` 方法将未来日期**回溯映射**到历史上最近可用的同星期数据（`minusDays(364)` 保持星期对齐），在 `date_mapping_temp` 临时表中存储映射关系，再调用 `generate_impression_report` 存储函数用历史数据"模拟"未来曝光。
- **投后（Post-campaign）**：投放周期已过，`actual_impression` 表中有真实数据。但不能直接使用原始值，会调用 `generate_impression_report_adjusted_total_impression`，对异常传感器数据和不完整执行周期进行修正（adjusted）。

**判定逻辑**：
```java
// ImpressionReportService.generateReport
LocalDate maxDate = officialDailyTrafficRepository.findMaxDate(envInfoCode);
boolean isPostCampaign = !endDate.isAfter(maxDate); // 结束日在已有数据范围内
```

**一致性风险**：投前报告的三次独立 `commit()`（request_temp → date_mapping_temp → 调用存储函数）存在**原子性缺失**风险——第二次 commit 后崩溃，临时表有数据但报告表没有写入，处于不一致状态。理想方案是三步合并为一个事务，加 `ON COMMIT DROP` 临时表选项，保证原子性。

## 维度二：日期映射递归算法设计（4题）

### Q4. 你的日期映射算法是怎么设计的？解决了什么问题？

**满分回答：**

**业务问题背景**：广告主想看"如果我在2025年10月投放，预计曝光量是多少"，但 Redshift 里最新的实际流量数据只到2025年8月。不能直接查不存在的数据，也不能随机取一个历史日期（季节性流量差异大）——解决方案是**找到历史上同类型日期的流量作为代理**。

**算法设计**：
```
用户请求: startDate=2025-10-01, endDate=2025-10-14
数据可用边界: lastEndDate=2025-08-31

setAvailableDates 递归逻辑:
  if endDate > lastEndDate:
      startDate -= 52周 (364天)
      endDate   -= 52週 (364天)
      递归调用
  → 映射到 2024-10-02 ~ 2024-10-15 (同星期)
```

**为什么用 364 天而非 365 天**：364 = 52×7，减去整数周后，星期几完全不变。周一还是周一，周末还是周末。交通流量最强的影响因素是星期几（工作日 vs 周末的客流量差异可达 40%），保持星期对齐比保持日历日期对齐更重要。用 365 天会导致每年漂移一天，5 年后周几完全错位。

**递归 vs 迭代的取舍**：当前源码用递归，优点是表达力强、代码简洁；缺点是没有终止保护。工程上更好的写法是改为 while 循环并加 `maxRetries=100` 的防死循环保护：
```java
int maxRetries = 100;
while (endDate.isAfter(lastEndDate) && maxRetries-- > 0) {
    startDate = startDate.minusWeeks(52);
    endDate   = endDate.minusWeeks(52);
}
if (maxRetries == 0) throw new IllegalStateException("Date mapping exceeded max retries");
```

### Q5. setAvailableDates、setLastAvailableDates、getRefDate 三种映射策略有什么区别？

**满分回答：**

三者解决**粒度不同的映射需求**：

| 方法 | 调用方 | 映射粒度 | 特点 |
|------|--------|----------|------|
| `setAvailableDates` | KPIsService（模拟计算） | 整段时间平移 | startDate 和 endDate 同步后退，保持周期长度不变 |
| `setLastAvailableDates` | KPIsService（最近可用） | 逐天独立映射 | 每一天单独做 while 循环回溯，各天结果可以映射到不同历史周期 |
| `getRefDate` | ImpressionReportService | 单日双向映射 | 额外支持**前向映射**——如果日期早于数据起始，则向前加 364 天 |

**为什么 ImpressionReport 需要前向映射**：假设数据窗口是 `2023-01-01 ~ 2025-08-31`，但用户请求的报告包含 2022-12-15（早于窗口起点）。此时不仅要向后回溯，还要向前推进，`getRefDate` 的 `plusDays(364)` 就处理了这个边界：
```java
if (dateRef.isBefore(firstStartDate)) {
    dateRef = dateRef.plusDays(364); // 前向推进
}
while (dateRef.isAfter(lastEndDate)) {
    dateRef = dateRef.minusDays(364); // 后向回溯
}
```

KPIsService 的"模拟"场景只会有未来日期，不会有过去日期，所以不需要前向映射；ImpressionReport 支持历史报告补录，必须处理两个方向。

### Q6. 如果递归导致 StackOverflow 或死循环，你怎么防御？

**满分回答：**

这道题考察防御性编程意识。

**StackOverflow 风险评估**：`setAvailableDates` 每次递归减少约 1 年，从 2050 年回溯到 2024 年最多 26 层。JVM 默认栈约 512KB，每帧约 1-2KB，26 层远不会溢出。**但真正的风险是极端配置错误**——如果 `lastEndDate` 因 Bug 被设为 1900 年，递归深度约 125 层，仍然安全；若设为 `LocalDate.MIN`，则理论上数百层，可能触发 SOF。

**死循环风险**：`setLastAvailableDates` 的 `while` 循环依赖 `DateUtils.getLastYearDate(dateRef)` 每次返回更早的日期。如果该方法 Bug 导致返回原值，while 永远为真，**线程永久阻塞**，最终导致线程池耗尽和 OOM。

**生产级防御方案**：
```java
// 递归改迭代 + 上限保护
public void setAvailableDates(KPIsQueryDto dto, LocalDate lastEndDate) {
    int maxIterations = 200;
    while (dto.getEndDate().isAfter(lastEndDate)) {
        if (--maxIterations == 0) {
            throw new IllegalStateException(
                "Date mapping loop exceeded limit. lastEndDate=" + lastEndDate 
                + ", endDate=" + dto.getEndDate());
        }
        LocalDate prev = dto.getStartDate();
        dto.setStartDate(dto.getStartDate().minusWeeks(52));
        dto.setEndDate(dto.getEndDate().minusWeeks(52));
        // 额外校验：确保日期确实在向前移动
        if (!dto.getStartDate().isBefore(prev)) {
            throw new IllegalStateException("getLastYearDate returned non-decreasing date");
        }
    }
}
```

### Q7. date_mapping_temp 临时表为什么在 Java 层预计算映射，而不是在 SQL 函数内部计算？

**满分回答：**

这是"计算层分工"的架构设计决策，核心理由有三：

**1. 测试性**：日期映射逻辑放在 Java 层可以写单元测试，直接断言每个输入日期的映射输出。PL/pgSQL 内部的 while 循环难以隔离测试，只能做集成测试，维护成本高。

**2. 调试性**：Java 层计算完后，可以在日志里打印完整的映射表（`original_date → mapping_date`），出问题时肉眼可见。SQL 函数内部的中间计算结果无法直接打印。

**3. 数据库负担**：映射计算本质上是纯内存的日期运算，没有 IO。放在 Java 层用 CPU 计算，远比在数据库中启动 PL/pgSQL 解释器执行 while 循环更高效，也减少了数据库连接的持有时间。

**两步 JOIN 设计**：`generate_impression_report` 函数通过 `date_mapping_temp.original_date BETWEEN start_date AND end_date` 展开日期范围，再用 `actual_impression.date = date_mapping_temp.mapping_date` 找真实数据，这种设计：
- 保持 SQL 函数职责单一（只做数据组装和写入）
- `date_mapping_temp` 可在同一 session 内复用（跨多次函数调用）
- PostgreSQL 的 CTE 优化器能将两步 JOIN 合并为高效的执行计划

## 维度三：差异化曝光计算引擎设计（5题）

### Q8. 数字媒体和传统媒体的曝光量计算为什么公式不同？LEAST(flow * sot, 1) 的含义是什么？

**满分回答：**

**物理场景差异**：
- **传统媒体**（灯箱、海报）：广告 24 小时静态展示，100% 时间都在曝光，所有经过的人都有机会看到。曝光 = 经过人数（traffic），无时间分摊问题。
- **数字媒体**（LED 大屏、数字广告牌）：多个广告轮播播出，SOT（Share of Time）= 该广告占屏幕播放时长的比例。SOT=0.3 意味着每 10 秒轮播周期里，该广告只展示 3 秒。只有在这 3 秒内经过的人才被"曝光"到。

**公式解析**：
```sql
-- 数字媒体
viewable_impression = LEAST(exp_otc_flow * sot, 1) * traffic + exp_otc_dwell * traffic

-- 传统媒体  
viewable_impression = traffic + exp_otc_dwell * traffic
```
- `exp_otc_flow`：经过受众中实际看到广告的概率系数（由视角、距离等决定）
- `exp_otc_dwell`：驻留受众（等车时停留）看到广告的额外曝光
- `LEAST(..., 1)` 的封顶：当 `exp_otc_flow * sot > 1` 时（多块屏幕同时展示同一广告或 SOT 很大），物理上不可能超过 100% 的人看到，封顶为 1 保证曝光量不超过经过总人数。这是**业务语义的保证**，不是数学上的可选优化。

**扩展性设计**：当前用 SQL 中的 `CASE asset.is_digital WHEN true THEN ... ELSE ...` 区分。新增第三种媒体类型时，可重构为策略模式——在 `asset` 表增加 `formula_type` 字段，Java 层通过工厂方法选择对应的计算策略类，SQL 只做数据聚合，计算逻辑在 Java 层实现，符合开闭原则。

### Q9. Population 多分段体系如何做到可配置？动态列名是怎么生成的？

**满分回答：**

**配置驱动的多分段**：`application-mam.properties` 中：
```properties
dc.generic.am.population={"1":"traffic","2":"traffic_departure"}
```
注入到 `ImpressionService.populations`（类型 `LinkedHashMap<Integer, String>`）。Key 是 population 编号，Value 是对应的 Redshift 基础列名。

**从配置到 SQL 的完整数据流**：
```
配置文件 population map
    ↓ @Value("#{...}") SpEL 注入
ImpressionService.populations = {1:"traffic", 2:"traffic_departure"}
    ↓ getImpressionsSumParam 遍历 map
Constants.TRAFFIC_POPULATION_COL.get(1) = "traffic"
Constants.VIEWED_IMPRESSIONS_COL.get(1) = "viewed_impressions"
    ↓ StringBuilder 拼接
SUM(traffic) as traffic_1, SUM(viewed_impressions) as viewed_1,
SUM(traffic_departure) as traffic_2, SUM(viewed_impressions_population_2) as viewed_2
    ↓ @Select + StatementType.STATEMENT
Redshift 执行动态 SQL
```

**硬编码 Map 的问题与改进**：`Constants` 中 8 个 `Map<Integer, String>` 硬编码了列名映射，违反 OCP——新增 population 要改 8 处。改进方案是规范化列名生成规则：
```java
public static String getTrafficColumn(int popKey) {
    if (popKey == 1) return "traffic";
    if (popKey == 2) return "traffic_departure";
    return "traffic_population_" + popKey;
}
```
通过命名规则动态推导列名，而非查表，新增 population 无需改代码。

### Q10. ImpressionService 中 DEFAULT_SOT=999.0 的特殊含义是什么？为什么不用 null？

**满分回答：**

**999.0 是哨兵值（Sentinel Value）**，代表"全量 SOT，不做 SOT 加权"——即传统媒体或用户没有指定 SOT 的场景，直接使用预聚合的 `viewed_impressions` 列，而非实时计算。

**两条 SQL 路径**：
```java
if (shareOfTime.equals(DEFAULT_SOT)) {
    // 路径一：直接 SUM(viewed_impressions)，用预聚合值，查询快
    sql = "SUM(viewed_impressions)";
} else {
    // 路径二：CASE WHEN 条件聚合，用原始流量数据实时计算
    sql = "SUM(CASE WHEN flow * sot > coefficient THEN ... ELSE ... END)";
}
```

**为什么不用 null 而用 999.0**：
1. `null` 在 Java 基本类型 `double` 中无法表示，需要装箱为 `Double`，且每处使用都需要 null check，代码更繁琐。
2. 999.0 在业务上不可能是合法的 SOT 值（SOT 范围 0.0-1.0），用它作哨兵值不会产生歧义。
3. 哨兵值适合作为配置项（`dc.generic.am.shareOfTime`），可以在不同平台配置不同的"默认 SOT 含义"。

**Trade-off**：哨兵值的缺点是隐式语义，代码可读性不如显式的 `Optional<Double>` 或枚举。如果接口文档不说明，新来的开发者看到 999.0 会困惑。更好的方案是定义 `enum SotMode { DEFAULT, CUSTOM }` + `Double sotValue`，使语义显式化。

### Q11. OnDemand 路径中虚拟 SOT 表 JOIN 的设计意图是什么？

**满分回答：**

**问题背景**：OnDemand 场景中，用户可能为 50 个 asset 分别配置不同的 SOT（如 asset1→0.1, asset2→0.4, asset3→0.3...）。如果为每个 (asset, sot) 组合发一次 Redshift 查询，50 个 asset 就是 50 次网络往返，延迟不可接受。

**虚拟 SOT 表方案**：在 Java 层拼接一个 UNION ALL 虚拟表，和 `daily_impressions` 做一次 JOIN，一次查询返回所有 asset 的计算结果：
```sql
SELECT d.asset_id, SUM(CASE WHEN d.flow * s.sot > coeff THEN ... END)
FROM daily_impressions d
JOIN (
    SELECT 'asset1' as asset_id, 0.1 as sot
    UNION ALL SELECT 'asset2', 0.4
    UNION ALL SELECT 'asset3', 0.3
    -- ... 50 个 asset
) s ON d.asset_id = s.asset_id
WHERE d.date IN (...)
GROUP BY d.asset_id
```

**性能对比**：50 个 asset × 3 种 SOT = 150 次独立查询 vs 1 次 JOIN 查询。Redshift 每次查询有约 100ms 的连接建立和解析开销，1 次 JOIN 总耗时 ~500ms，150 次查询总耗时 ~15s——差异约 30 倍。

**风险控制**：当 asset 数量达到 500 个时，UNION ALL 生成约 `500 * len("SELECT 'assetXXXX', 0.XX UNION ALL ")` ≈ 30KB 的 SQL 文本。Redshift 的 SQL 解析器对 30KB 没有问题（上限约 16MB），但应在业务层做 asset 数量上限检查（如 1000 个），避免极端 SQL 膨胀。

### Q12. Redshift 的 MyBatis `${}` 字符串替换存在 SQL 注入风险吗？如何防御？

**满分回答：**

**风险评估**：项目中 `${}` 替换的变量来源有两类：

1. **动态列名**（如 `${trafficColumn}`）：列名来自 `Constants` 中的 Map，Map 的值是代码中硬编码的字符串，不来自用户输入。**无注入风险**。

2. **日期范围字符串**（如 `${dateRange}`）：由 `XAMUtils.getDateRangesFilters` 从 `List<LocalDate>` 拼接，`LocalDate.toString()` 只输出 `yyyy-MM-dd` 格式。**正常情况无注入风险**。但如果攻击者通过反序列化漏洞注入了恶意 LocalDate 子类（重写 `toString()` 返回 SQL 片段），理论上有风险。

3. **profiles 字符串**（`${profiles}`）：经过了 9 次 `REGEXP_REPLACE` 处理 URL 编码字符，但**没有转义单引号**，存在潜在风险。

**防御方案**：
```java
// 白名单校验动态列名
private static final Set<String> ALLOWED_COLUMNS = Set.of(
    "traffic", "traffic_departure", "viewed_impressions", "viewable_impressions"
    // ... 完整白名单
);
public static void validateColumn(String col) {
    if (!ALLOWED_COLUMNS.contains(col)) throw new IllegalArgumentException("非法列名: " + col);
}

// 校验 profile 不含特殊字符
private static final Pattern SAFE_PROFILE = Pattern.compile("^[a-zA-Z0-9_\\-@.]+$");
```

**为什么不改用 `#{}`（PreparedStatement）**：Redshift 的 `PreparedStatement` 不支持将**标识符**（列名、表名）作为参数绑定，`?` 只能绑定值。动态列名场景只能用字符串拼接，因此防御的核心是白名单校验，而非参数化查询。

## 维度四：CompletableFuture 异步并行实战（5题）

### Q13. 从 8 秒优化到 1.5 秒，你是怎么分析瓶颈、设计并行方案的？

**满分回答：**

**瓶颈分析**：原始同步调用链为：
```
getHourlyImpression (2s)  
→ getImpression (2s)  
→ getDailyImpression (1.5s)  
→ getTotalTraffic (1s)  
→ getReach (1.5s)  
= 串行总计 8s
```
通过日志加时间戳发现，5 个 Redshift 查询之间没有数据依赖（各自查询不同的聚合视图），完全可以并行。

**并行方案**：在 `KPIsService.setData` 中，用 `CompletableFuture.supplyAsync` 将 5-6 个查询提交到线程池并行执行：
```java
ExecutorService executor = Executors.newFixedThreadPool(10);

CompletableFuture<List<ImpressionResponseDto>> impressionFuture = 
    CompletableFuture.supplyAsync(() -> impressionService.getImpression(...), executor);

CompletableFuture<List<TrafficResponseDto>> trafficFuture = 
    CompletableFuture.supplyAsync(() -> trafficService.getTotalTraffic(...), executor);
// ... 其余任务同理并行提交
```
最长的单个查询约 1.5s，并行后总耗时 ≈ 1.5s（而非串行 8s）。

**线程池配置考量**：IO 密集型任务的推荐线程数 = `CPU核心数 × (1 + IO等待时间/CPU计算时间)`。Redshift 查询 RT 约 1-2s，CPU 计算可忽略，IO 等待比约 100:1。4 核机器理论值约 400 线程，但过多线程会引起上下文切换开销，实际取 10 线程是经验性保守值，可根据线上监控调整。

### Q14. 你的代码中 `AtomicReference<CompletableFuture>` + `Consumer::set` 是什么设计？为什么这样做？

**满分回答：**

**问题根源**：Java lambda 要求捕获的局部变量必须是 `effectively final`——即一旦赋值就不能修改引用。但 `CompletableFuture` 是在 lambda 内部创建的，需要把引用"传出"给外部代码，普通局部变量做不到。

**AtomicReference 的绕过方式**：
```java
AtomicReference<CompletableFuture<List<ImpressionResponseDto>>> impressions = new AtomicReference<>();

// setData 接受一个 Consumer，内部完成 Future 创建后调用 consumer.accept(future)
kPIsService.setData(dto, response, impressions::set, /* 其他参数 */);

// 此时 impressions.get() 已拿到 Future 引用
List<ImpressionResponseDto> result = impressions.get().get();
```
`AtomicReference` 本身是 effectively final（引用不变），但其内部值可变，通过 `set/get` 方法绕过了限制。

**更优雅的替代方案**：
```java
// 方案一：setData 直接返回 CompletableFuture
public CompletableFuture<KPIsDto> setData(KPIsQueryDto dto) {
    CompletableFuture<List<ImpressionResponseDto>> imp = CompletableFuture.supplyAsync(...);
    CompletableFuture<List<TrafficResponseDto>> traf = CompletableFuture.supplyAsync(...);
    return CompletableFuture.allOf(imp, traf)
        .thenApply(v -> buildKpisDto(imp.join(), traf.join()));
}

// 方案二：返回 Map<String, CompletableFuture<?>>，key 为指标名称
```
直接返回 Future 更符合函数式设计，避免了 AtomicReference 的"魔法"写法。

### Q15. CompletableFuture 某个子任务失败，整个链路怎么兜底？

**满分回答：**

**当前代码的问题**：`setOverAllKpis` 中用 `impressions.get()` 阻塞获取结果，如果任何一个 `Future.get()` 抛出 `ExecutionException`，catch 块直接标记整个 activity 为 FAILED，**已成功的其他查询结果全部丢弃**。

**生产级兜底方案**——每个 Future 独立降级：
```java
// 给每个 Future 加降级处理，部分失败不影响其他模块
CompletableFuture<List<ImpressionResponseDto>> safeImpressions = impressionFuture
    .exceptionally(ex -> {
        log.error("Impression query failed, returning empty", ex);
        metrics.increment("kpis.impression.failure");
        return Collections.emptyList(); // 降级返回空
    });

CompletableFuture<List<TrafficResponseDto>> safeTraffic = trafficFuture
    .exceptionally(ex -> {
        log.error("Traffic query failed, returning empty", ex);
        return Collections.emptyList();
    });

// 用 allOf 等所有任务完成后再聚合
CompletableFuture.allOf(safeImpressions, safeTraffic, ...)
    .thenAccept(v -> buildResponse(safeImpressions.join(), safeTraffic.join(), ...));
```

**超时控制**：生产环境 Redshift 有时会因负载高导致查询超时：
```java
CompletableFuture<List<ImpressionResponseDto>> withTimeout = impressionFuture
    .orTimeout(10, TimeUnit.SECONDS)
    .exceptionally(ex -> {
        if (ex instanceof TimeoutException) log.warn("Impression query timed out");
        return Collections.emptyList();
    });
```
`orTimeout` 是 Java 9+ 的方法，超时后自动触发 `CancellationException` → `exceptionally` 降级。

### Q16. 你的线程池是每次请求新建的，这有什么问题？应该怎么改？

**满分回答：**

**严重问题**：源码中 `Executors.newFixedThreadPool(10)` 在 `setData` 方法内部每次调用都新建，且**从不 shutdown**。这会导致：
1. **线程泄漏**：每个请求创建 10 个线程，100 QPS × 10 = 1000 线程/秒，1 分钟内 60,000 线程，耗尽 JVM 线程数限制（默认 `-Xss` 每线程约 1MB，1000 线程 = 1GB 栈内存）。
2. **JVM 无法正常退出**：`newFixedThreadPool` 创建的是非 daemon 线程，会阻止 JVM shutdown hook 执行，导致 `kill` 信号后服务无法优雅停止。

**正确方案——Spring Bean 共享线程池**：
```java
@Configuration
public class AsyncConfig {
    @Bean(name = "kpiExecutor", destroyMethod = "shutdown")
    public ExecutorService kpiExecutor() {
        return new ThreadPoolExecutor(
            10,                              // 核心线程数
            50,                              // 最大线程数（应对突发）
            60L, TimeUnit.SECONDS,           // 空闲线程存活时间
            new LinkedBlockingQueue<>(200),  // 有界队列，防 OOM
            new ThreadFactory() {
                private final AtomicInteger count = new AtomicInteger();
                public Thread newThread(Runnable r) {
                    Thread t = new Thread(r, "kpi-executor-" + count.incrementAndGet());
                    t.setDaemon(true); // daemon 线程，不阻碍 JVM 退出
                    return t;
                }
            },
            new ThreadPoolExecutor.CallerRunsPolicy() // 队列满时由调用者线程执行，自然限流
        );
    }
}
```
`destroyMethod = "shutdown"` 确保 Spring 容器关闭时线程池优雅关闭，等待当前任务完成后停止。

### Q17. @Cacheable 和异步线程能正常配合吗？会有缓存击穿问题吗？

**满分回答：**

**@Cacheable 在异步线程中是否生效**：**生效**。`@Cacheable` 基于 Spring AOP 动态代理——异步线程中调用的是从 Spring 容器注入的代理对象（`impressionService` Bean），AOP 切面照常拦截，缓存逻辑正常工作。**不生效的唯一场景**是同类内部自调用（`this.method()`），因为绕过了代理。

**缓存击穿风险**：项目中 `@Cacheable` 没有配置 `sync = true`。当 Redis 缓存失效瞬间，如果有 10 个并发请求同时 cache miss，全部去 Redshift 查询，产生 10 倍查询压力——这就是**缓存击穿**。

**解决方案**：
```java
// 方案一：sync=true（单 JVM 串行加载，适合非集群场景）
@Cacheable(value = "impressions", keyGenerator = "customKeyGenerator", sync = true)
public List<ImpressionResponseDto> getImpression(...) { ... }

// 方案二：分布式锁（Redisson）
public List<ImpressionResponseDto> getImpression(...) {
    String key = buildCacheKey(...);
    String cached = redis.get(key);
    if (cached != null) return deserialize(cached);
    
    RLock lock = redisson.getLock("lock:" + key);
    try {
        lock.lock(5, TimeUnit.SECONDS);
        // double check
        cached = redis.get(key);
        if (cached != null) return deserialize(cached);
        List<ImpressionResponseDto> result = impressionRepository.query(...);
        redis.setex(key, 3600, serialize(result));
        return result;
    } finally {
        lock.unlock();
    }
}
```

另外注意**缓存雪崩**（大量 key 同时过期）：项目的 Redis key 应加随机过期时间偏移（如 `3600 + random(0, 300)` 秒），避免同一时刻批量过期。

## 维度五：Apache POI 与 PPT 生成（3题）

### Q18. 大批量数据渲染 PPT 时如何避免 JVM OOM？

**满分回答：**

**内存峰值分析**：`XMLSlideShow` 将整个 PPT 以 XML DOM 树形式加载到堆内存。50 页幻灯片，每页含图表（EmbeddedXls 数据）+ 表格（42 cells）+ 背景图，估算单份 PPT DOM 约 30-80MB。`ppt.write(byteArrayOutputStream)` 时 `ByteArrayOutputStream` 内部缓冲区扩容策略是**每次翻倍**：写入 50MB 数据时，最坏情况旧缓冲 32MB + 新缓冲 64MB，再加 `toByteArray()` 复制一份 50MB，**峰值三倍内存**约 150-240MB。5 个并发请求同时生成则 750MB+，在 512MB 堆下必然 OOM。

**优化方案一：直接写入临时文件，绕过 ByteArrayOutputStream**：
```java
Path tempFile = Files.createTempFile("report-", ".pptx");
try (OutputStream os = Files.newOutputStream(tempFile)) {
    ppt.write(os); // 流式写入文件，不占用堆内存
}
ppt.close(); // 尽早释放 DOM 内存
byte[] bytes = Files.readAllBytes(tempFile); // 仅在必须时读入
```

**优化方案二：Logo 图片限制分辨率**：S3 下载的客户 Logo 可能是高分辨率 PNG，解码为 `BufferedImage` 峰值达 48MB（4000×3000×4B）。应在插入前缩放：
```java
BufferedImage original = ImageIO.read(inputStream);
// 限制最大宽度为 400px
BufferedImage scaled = Scalr.resize(original, Scalr.Method.BALANCED, 400);
original.flush(); // 立即释放原始大图
byte[] pngBytes = toBytes(scaled);
ppt.addPicture(pngBytes, PictureData.PictureType.PNG);
```

**优化方案三：并发控制**——用 Semaphore 限制同时生成的报告数：
```java
private final Semaphore reportSemaphore = new Semaphore(3); // 最多 3 个并发

public byte[] generateReport(...) {
    reportSemaphore.acquire();
    try {
        return doGenerate(...);
    } finally {
        reportSemaphore.release();
    }
}
```

### Q19. PPT 模板驱动生成的核心机制是什么？如果模板设计师改了 Layout 名，怎么感知？

**满分回答：**

**模板驱动机制**：`PPTUtils.generateImpressionReport` 通过 Apache POI 的 `XSLFSlideLayout` 名称（如 "Cover Shared"、"Info Page"、"Chart Page"、"Table Page"、"Last Page"）发现模板中的页面布局，然后 `XSLFSlide slide = ppt.createSlide(layout)` 创建对应页面，再用 `slide.getPlaceholder(idx)` 按占位符索引填充内容：
```java
// 按 Layout 名查找布局
XSLFSlideLayout layout = slideLayouts.stream()
    .filter(l -> l.getName().equals("Chart Page"))
    .findFirst()
    .orElseThrow(() -> new IllegalStateException("Layout 'Chart Page' not found in template"));

XSLFSlide slide = ppt.createSlide(layout);
// 按占位符索引填充
XSLFTextShape title = (XSLFTextShape) slide.getPlaceholder(0);
title.setText(pageTitle);
```

**Layout 名修改的感知**：当前代码**不会在编译期感知**，只会在运行时抛 `IllegalStateException` 或 `NullPointerException`（如果用 `findFirst().orElse(null)` 后没判空）。这是"约定式"（Convention-based）设计的固有缺点。

**改进方案**：
1. **启动时校验**：用 `@PostConstruct` 在应用启动时加载模板并校验所有必需 Layout 名称是否存在，启动失败比运行时失败更友好。
2. **枚举化 Layout 名**：将 Layout 名定义为 `enum LayoutType { COVER("Cover Shared"), CHART("Chart Page"), ... }`，模板更换时改枚举一处即可，消除字符串魔法值。
3. **集成测试**：对模板文件写一个集成测试，断言所有枚举值都能在模板中找到对应 Layout。

### Q20. PPT 转 PDF 为什么要经过 soffice（LibreOffice）？为什么不直接生成 PDF？

**满分回答：**

**为什么不直接生成 PDF**：Apache POI 本身**不支持直接输出 PDF**——POI 只处理 Office Open XML 格式（.pptx/.xlsx/.docx）。Java 生态中没有成熟的免费库能将复杂 PPT（带自定义字体、图表、占位符动画）高保真地直接渲染为 PDF。Apache PDFBox 只处理 PDF 读写，不做 PPT→PDF 转换。

**LibreOffice 的优势**：`soffice --headless --convert-to pdf` 使用 LibreOffice 的完整排版引擎，能处理自定义字体、嵌入图表和复杂布局，PDF 还原度高，且是免费开源的。业务上客户对 PDF 样式有严格要求，LibreOffice 是目前最可靠的方案。

**当前代码的风险与改进**：
```java
// 当前代码：没有检查退出码，没有清理临时文件
Process process = runtime.exec(sofficePath + " --headless --convert-to pdf ...");
process.waitFor();
byte[] pdfBytes = Files.readAllBytes(Paths.get(tempPDF)); // 如果 soffice 崩溃，这里 NPE

// 改进版本
Path tempDir = Files.createTempDirectory("ppt2pdf");
try {
    ProcessBuilder pb = new ProcessBuilder(
        sofficePath, "--headless", "--convert-to", "pdf",
        "--outdir", tempDir.toString(), pptxPath.toString()
    );
    pb.redirectErrorStream(true);
    Process p = pb.start();
    boolean finished = p.waitFor(30, TimeUnit.SECONDS);
    if (!finished || p.exitValue() != 0) {
        String output = new String(p.getInputStream().readAllBytes());
        throw new RuntimeException("soffice failed: " + output);
    }
    return Files.readAllBytes(tempDir.resolve("output.pdf"));
} finally {
    // 无论成功失败，清理临时目录防止磁盘泄漏
    Files.walk(tempDir).sorted(Comparator.reverseOrder())
         .forEach(p -> p.toFile().delete());
}
```
重点改进：超时控制（30s）、退出码检查、`finally` 块清理临时文件防磁盘泄漏。

| 问题 | 一句话核心要点 |
|------|---------------|
| `HashMap` 为什么线程不安全？ | JDK8 前并发 resize 会形成链表环导致死循环；JDK8 后并发 put 可能数据覆盖 |
| `ConcurrentHashMap` JDK8 的锁粒度 | CAS + synchronized 锁单个桶（Node），不锁整个 table |
