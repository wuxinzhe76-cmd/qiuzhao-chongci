### 1. 限流：本地令牌桶 -> 分布式限流

| 补充建议 | 评估 | 德高是否需要 |
|---------|------|------------|
| 令牌桶没补充令牌（bug） | ✅ 正确，代码确实有bug | **必须修复** |
| 不支持并发 | ✅ 正确，需要加锁 | **必须修复** |
| 改为Redis分布式限流 | ⚠️ 部分采纳 | **单机部署不需要，但代码bug必须修** |
| 多维度限流（租户/用户/接口） | ⚠️ 适度采纳 | **按用户和接口限流够用** |
| 排队和背压 | ⚠️ 适度采纳 | **简单优先级队列够用** |

**德高实际情况**：
- 单机部署，FastAPI + uvicorn 多 worker，不需要分布式限流
- 但令牌桶代码的bug必须修复（补充令牌 + 加锁）
- 50-100人同时使用，本地令牌桶够用
- 如果未来扩到多实例，再上Redis分布式限流

**采纳方案**：
```python
import time
import threading

class TokenBucket:
    """修复版令牌桶：补充令牌 + 线程安全"""
    
    def __init__(self, capacity=10, refill_rate=10):
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.tokens = capacity
        self.last_refill = time.time()
        self.lock = threading.Lock()  # 加锁防止并发超发
    
    def acquire(self) -> bool:
        with self.lock:
            # 补充令牌（原代码缺这一步）
            now = time.time()
            elapsed = now - self.last_refill
            self.tokens = min(self.capacity, self.tokens + elapsed * self.refill_rate)
            self.last_refill = now
            
            if self.tokens >= 1:
                self.tokens -= 1
                return True
            return False

# 多维度限流（适度采纳）
RATE_LIMIT = {
    "per_user": {"capacity": 30, "refill_rate": 0.5},  # 每人每分钟30次
    "per_api": {
        "chat": {"capacity": 50, "refill_rate": 50},     # 问答50 QPS
        "video": {"capacity": 5, "refill_rate": 1},       # 视频生成5并发
        "report": {"capacity": 10, "refill_rate": 10}     # 报告生成10 QPS
    }
}
```

**德高实际情况**：

- 如果缓存不按权限隔离，上海销售可能看到北京的数据

**采纳方案**：
```python
# 缓存键必须包含权限信息
def build_cache_key(query, user_context):
    return hashlib.md5(f"""
        {user_context.tenant_id}
        {user_context.department_id}  # 城市分公司
        {user_context.role}
        {query}
        {get_index_version()}  # 索引版本
    """.encode()).hexdigest()

# 权限隔离的语义缓存
def semantic_cache_get(query, user_context):
    # 向量检索时必须带ACL过滤
    results = milvus.search(
        collection="query_cache",
        data=[embed(query)],
        top_k=1,
        filter=f"department_id == '{user_context.department_id}'",  # 权限过滤
        output_fields=["query", "answer"]
    )
    if results and results[0].score >= 0.90:  # 阈值不写死0.95
        return results[0].answer
    return None
```

def reconcile_neo4j_milvus():

# Cypher查询加权限过滤
def graph_traverse_with_acl(entity_name, user_context, hops=2):
    cypher = f"""
    MATCH path = (n {{name: $name}})-[*1..{hops}]-(related)
    WHERE related.department_id = $dept_id  // 权限过滤
      AND related.classification IN $allowed_classifications
    RETURN nodes(path), relationships(path)
    LIMIT 50
    """
    return neo4j.run(cypher, 
                     name=entity_name, 
                     dept_id=user_context.department_id,
                     allowed_classifications=user_context.allowed_classifications)

# Milvus检索加权限过滤
def vector_search_with_acl(query, user_context, top_k=10):
    return milvus.search(
        collection="text_chunks",
        data=[embed(query)],
        top_k=top_k,
        filter=f"department_id == '{user_context.department_id}'",  // 权限过滤

## 二、总结：采纳 / 适度采纳 / 过度设计

### 必须采纳（8项）

| 编号 | 补充建议 | 原因 |
|------|---------|------|
| 1 | 令牌桶bug修复（补充令牌+加锁） | 代码确实有bug |
| 2 | 组件级熔断+降级矩阵 | 不同组件故障降级方案不同 |
| 3 | 可重试/不可重试错误分类 | 避免重试不可恢复的错误 |
| 4 | 幂等键（视频生成） | 避免重复生成、重复计费 |
| 5 | 缓存权限隔离 | 防止A部门数据泄露给B部门 |
| 6 | asyncio.create_task改Celery | 进程崩溃不丢任务 |
| 7 | OpenTelemetry链路追踪 | 排查哪段慢 |
| 8 | GraphRAG专属监控指标 | 检索质量/数据质量/LLM质量 |

### 适度采纳（5项）

| 编号 | 补充建议 | 德高方案 |
|------|---------|---------|
| 1 | 多维度限流 | 按用户+接口限流，不需要租户/Token/成本限流 |
| 2 | 优先级队列 | 简单P0-P5优先级，不需要完整Bulkhead |
| 3 | 索引版本化 | 版本号+对账任务，不需要蓝绿索引 |
| 4 | 权限控制 | 按城市/部门隔离，不需要细粒度ACL |
| 5 | 分阶段超时 | 首Token<3秒，不需要全部P95<3秒 |

### 过度设计（德高规模不需要）

| 编号 | 补充建议 | 不需要的原因 |
|------|---------|------------|
| 1 | Redis分布式限流 | 单机部署，本地令牌桶够用 |
| 2 | Temporal工作流引擎 | Celery足够，Temporal太重 |
| 3 | 完整Bulkhead隔舱 | 1000人规模不需要独立连接池/独立资源配额 |
| 4 | 蓝绿索引 | 5-10万节点，简单版本号够用 |
| 5 | 完整八部分重构 | 保留6部分+补充关键点即可 |
