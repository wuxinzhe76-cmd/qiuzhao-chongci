### Q6. setAvailableDates vs setLastAvailableDates vs getRefDate 三种映射策略的差异

请回答：

1. `setAvailableDates` 是将整个日期范围整体平移（startDate 和 endDate 同时回退），而 `setLastAvailableDates` 是逐天独立映射。这两种策略分别用在什么业务场景？为什么 KPIs 模拟用整体平移，而 ImpressionReport 用逐天映射？
2. `getRefDate` 相比前两个方法多了一个**前向映射**能力（当日期早于数据窗口时，向前加 364 天）。为什么 ImpressionReport 需要前向映射？举一个具体场景说明。

### Q7. 为什么用 364 天（52周）而不是 365 天作为年度回溯步长？

`DateUtils.getLastYearDate` 的实现是 `date.minusWeeks(52)` = 减去 364 天。`ImpressionReportService.getRefDate` 用 `plusDays(364)` / `minusDays(364)`。请回答：

1. 选择 364 天而非 365 天的核心原因是什么？这个选择保证了什么不变量？
2. 364 天偏移会导致日期"漂移"（如 1月1日 回退5年后变成 12月27日）。在业务上，周几对交通流量的影响远大于日历日期的精确对齐——请解释这个设计取舍的逻辑。

### Q8. date_mapping_temp 临时表在 SQL 函数中的作用

`ImpressionReportService.generateReport` 方法先在 Java 层计算好每天对应的映射日期，写入 `date_mapping_temp` 临时表，再调用 `generate_impression_report` SQL 函数。请回答：

1. 为什么不在 SQL 函数内部直接计算映射日期（比如写一个 PL/pgSQL 的 while 循环），而是选择在 Java 层计算后传入？这种"Java 计算映射 + SQL 使用映射"的分工设计有什么好处？
2. `generate_impression_report` 函数通过 `date_mapping_temp.original_date BETWEEN request_temp.start_date AND request_temp.end_date` 来展开日期范围，然后用 `actual_impression.date = date_mapping_temp.mapping_date` 来查找曝光数据。这个两步 JOIN 的设计为什么不用子查询实现？

### Q11. 数字媒体 vs 传统媒体的曝光量计算公式差异

1. 从广告展示的物理场景出发，解释为什么数字媒体需要 SOT 系数而传统媒体不需要。`exp_otc_flow`（经过曝光）和 `exp_otc_dwell`（驻留曝光）分别代表什么？
