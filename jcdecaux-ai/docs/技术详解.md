### 为什么是两个独立系统？

1. **前端完全不同**：报告系统是数据表格+图表；AI 系统有"选拍摄角度/车型/城市"的下拉框，是视频生成界面
2. **技术栈不同**：报告系统是纯 Java；AI 系统多了大模型、知识图谱、Python 图像处理
3. **用户场景不同**：报告系统是"投后展示数据给客户看"；AI 系统是"投前帮销售决定该跟谁、怎么做提案"

### 两个系统的数据关联

AI 系统的知识图谱数据**部分来源于** generic-am-back 的业务数据：

```
generic-am-back 的数据         →    AI 系统的知识图谱节点
─────────────────────────         ─────────────────────────
Asset（广告位）                  →   点位节点（地铁站/公交站牌）
Activity（投放活动）              →   Campaign 节点（历史投放案例）
Network（线路网络）               →   City 的子结构
Redshift 流量数据                →   StoreMetric 节点（人流量指标）
Impression（曝光数据）            →   投放效果数据
```

## 三、决策引擎：GraphRAG（最核心，面试必问）

### 3.1 为什么不用普通 RAG？

普通 RAG（Retrieval-Augmented Generation）：
```
用户提问 → 向量数据库搜相似文档 → 把文档塞给大模型 → 大模型回答

问题：只靠"语义相似"找文档，缺乏业务逻辑关联
```

**举例说明普通 RAG 的缺陷：**

```
销售问："XX茶饮本月该跟吗？"

普通 RAG 检索到：
- "XX茶饮是奶茶品牌"（语义相似，但没用）
- "奶茶行业报告"（相关，但不具体）

它找不到这种逻辑链：
XX茶饮 → 杭州有12家门店 → 坪效环比+3% → 同类品牌在扩张期户外投放+2.1倍
```

普通 RAG 只能找"长得像"的内容，找不到"A→B→C→D"这种**多跳推理链**。

### 3.2 GraphRAG 怎么解决？

GraphRAG = **知识图谱 + 向量检索 + 社区摘要**

```
                    GraphRAG 检索过程

用户提问
   │
   ├─── 路径1：向量检索（找语义相似的节点）
   │         "XX茶饮" → 匹配到 Brand 节点
   │
   ├─── 路径2：图遍历（沿关系边多跳推理）
   │         Brand(XX茶饮)
   │           →OPERATES_IN→ City(杭州)
   │           →HAS_METRIC→ StoreMetric(坪效+3%)
   │           →SIMILAR_TO→ Campaign(某竞品广告案例)
   │           →LAUNCHED→ 历史投放数据
   │
   └─── 融合两路结果 → 送给大模型生成回答
```

### 3.3 知识图谱设计

```
节点（Node）类型：
┌─────────────┬──────────────────────────┐
│ Brand       │ 品牌（XX茶饮、某汽车）      │
│ City        │ 城市（杭州、上海）          │
│ StoreMetric │ 门店指标（坪效、店效、门店数）│
│ Campaign    │ 历史广告投放案例            │
│ Report      │ 行业报告                    │
│ Industry    │ 行业（快消、美妆、汽车）     │
└─────────────┴──────────────────────────┘

边（Edge/关系）类型：
┌─────────────────┬─────────────────────────────────┐
│ OPERATES_IN     │ Brand → City（品牌在某城市有门店）  │
│ HAS_METRIC      │ Brand → StoreMetric（品牌的门店指标）│
│ LAUNCHED        │ Brand → Campaign（品牌投放过广告）   │
│ SIMILAR_TO      │ Campaign → Campaign（相似广告案例）  │
│ BELONGS_TO      │ Brand → Industry（品牌属于某行业）   │
└─────────────────┴─────────────────────────────────┘
```

**和 generic-am-back 的关联：**

已有的投前投后系统里已经有这些数据了：
- `Asset`（广告位）= 知识图谱里的点位
- `Activity`（活动/投放）= Campaign 节点
- `Network`（线路网络）= City 的子结构
- Redshift 里的流量数据 = StoreMetric

所以 AI 项目的知识图谱**是基于已有系统的数据构建的**，不是从零开始。

### 3.4 Leiden 社区发现算法（面试重点）

**问题：** 图谱有几千个节点，每次查询都遍历全部太慢。

**解决：** 用 Leiden 算法把图谱**分层聚类**，提前算好每个社区的摘要。

```
聚类前（散乱的节点）：
  品牌A ── 城市1
  品牌B ── 城市1
  品牌C ── 城市2
  品牌D ── 城市2
  品牌E ── 城市3
  ...（几千个）

聚类后（分成社区）：
  社区1："华东快消圈"
    ├── 品牌A（茶饮）→ 杭州 → 坪效+3%
    ├── 品牌B（零食）→ 上海 → 坪效+1%
    └── 社区摘要："华东快消品牌整体扩张中，户外预算释放概率高"

  社区2："华南汽车圈"
    ├── 品牌C（新能源）→ 深圳 → 展厅+5家
    └── 社区摘要："华南新能源展厅扩张，新车上市周期"
```

**检索时：**
1. 先定位到属于哪个社区（快，只查社区摘要）
2. 再在社区内部做精细检索（节点少，快）

```
效果：检索延迟下降 60%
原因：不用遍历全图，只看目标社区
```

### 3.5 数据从哪来？怎么抽取？

```
数据源                          抽取方式
┌─────────────┐                ┌─────────────────────────┐
│ 电商数据     │  ──────────→   │ 结构化：直接映射成节点/边  │
│ (淘/京/抖)  │                │ brand_sales_monthly 等   │
└─────────────┘                └─────────────────────────┘
┌─────────────┐                ┌─────────────────────────┐
│ 门店效能     │  ──────────→   │ 结构化：直接映射          │
│ (店效/坪效)  │                │ store_metric 节点        │
└─────────────┘                └─────────────────────────┘
┌─────────────┐                ┌─────────────────────────┐
│ 历史案例     │  ──────────→   │ 半结构化：LLM + 规则抽取  │
│ (合同/PPT)  │                │ "品牌X在Y市投了Z类型广告"  │
└─────────────┘                │ → 提取三元组存入图谱      │
                               └─────────────────────────┘
┌─────────────┐                ┌─────────────────────────┐
│ 行业报告     │  ──────────→   │ 非结构化：GraphRAG       │
│ (PDF/Word)  │                │ pipeline 自动抽取实体关系 │
└─────────────┘                └─────────────────────────┘
```

**三元组抽取举例：**

```
输入文本："2024年5月，XX茶饮在杭州地铁投放了为期1个月的灯箱广告，
         投放后门店客流提升15%。"

LLM 抽取出的三元组：
(XX茶饮) -[LAUNCHED]-> (杭州地铁灯箱广告202405)
(杭州地铁灯箱广告202405) -[LOCATED_IN]-> (杭州)
(杭州地铁灯箱广告202405) -[DURATION]-> (1个月)
(XX茶饮) -[HAS_RESULT]-> (客流+15%)
```

## 四、执行引擎：视频生成工作流

### 4.1 为什么需要这个？

投前投后系统已经能生成 **PPT 报告**了（`PPTUtils.generateImpressionReport`）。
但 PPT 是"纸面方案"，销售见客户时更想展示"广告贴在车上/地铁站里实际长什么样"。

```
原来（已有系统）：生成 PPT 报告 → 销售拿 PPT 去谈客户
现在（AI系统）：直接生成"广告效果图+视频" → 客户直观看到效果
```

### 4.2 五步生成流程

```
第1步：素材抓取（0素材起步）
  输入：品牌名
  操作：Python 从公网抓取品牌 Logo/产品图
  技术：rembg（AI抠图）+ OpenCV（色彩规范化）
  输出：干净的透明背景品牌素材

  类比已有系统中的：S3Service 下载客户 Logo
  → ImpressionReportFeatureService 里的 S3 logo 下载逻辑

第2步：AI 概念图生成
  输入：品牌素材 + 风格参数
  操作：ComfyUI/SDXL 生成广告概念图
  技术：Stable Diffusion + ControlNet（约束构图尺寸）
  输出：3版广告概念图供选择

第3步：Python 精准贴合
  输入：概念图 + 车型 + 广告位
  操作：把广告图贴合到公交车/地铁的真实照片上
  技术：OpenCV 透视变换 + UV网格曲面校正
  输出：广告贴在车上的效果图

第4步：城市地标合成
  输入：贴好广告的车图 + 城市地标
  操作：把车放到城市背景中
  技术：光影匹配 + 比例校准
  输出：广告车在城市中的实景图

第5步：视频生成
  输入：合成图 + 运镜参数
  操作：生成动态视频
  技术：Doubao 1.6 / 可灵 API + 结构化 Prompt
  输出：2分钟出片
```

### 4.3 Java 后端怎么编排这个流程？

面试 Java 岗位时要重点讲：

```
销售点击"生成视频"
    │
    ▼
Java 后端接收请求
    │
    ├── 创建异步任务（不能让用户等2分钟）
    │   → 类比已有系统的 OnDemand KPIs 异步计算
    │   → OndemandKpisService.calculateKPIsOnDemandCampaign
    │
    ├── 步骤1-4：调用 Python 服务（HTTP/RPC）
    │   → 类比已有系统调用 Livy Spark 任务
    │   → 提交 → 轮询 → 回调
    │
    ├── 步骤5：调用视频生成 API
    │   → 类比已有系统调用 S3/SQS
    │
    └── 结果通知（SSE 流式推送给前端）
        → 类比已有系统的异步任务状态轮询
```

**和已有系统的对应关系：**

| AI项目 | generic-am-back |
|--------|---------------------|
| 长耗时视频生成任务 | OnDemand KPIs 异步计算 |
| 轮询 Python 服务状态 | 轮询 Livy Spark Job 状态 |
| SSE 流式通知前端 | Job 表状态轮询 |
| 临时数据7天过期 | ImpressionReport 数据管理 |
| S3 存储生成结果 | S3 存储 PPT/PDF |

## 五、融合中枢：Agent（面试重点）

### 5.1 Agent 是什么？

Agent = **能自己决定调用哪个工具的大模型**

```
普通大模型：你问什么它答什么（只能说话，不能干活）
Agent：你问什么，它自己决定该查数据库还是查舆情还是生成视频（能干活）
```

### 5.2 在这个项目里 Agent 干什么？

```
销售问："本月杭州区的XX茶饮该主动去推吗？"

Agent 的思考过程：
1. 分析问题 → 需要历史数据 + 实时信号
2. 调用工具1：GraphRAG 检索
   → 结果：XX茶饮杭州门店+12家，坪效+3%，扩张期
3. 调用工具2：博查API查实时舆情
   → 结果：今天新华社报道XX茶饮签约顶流代言人
4. 调用工具3：查相似案例
   → 结果：同类品牌"扩张+代言"双信号下，户外投放+2.1倍
5. 融合判断 → 结论：建议主动触达，置信度92%
6. 生成自然语言回复 + 推荐动作
```

### 5.3 Function Calling 机制

```
Agent 通过 Function Calling 调用工具：

大模型收到问题
    │
    ▼
大模型判断："我需要调用 graphrag_search 工具"
    │
    ▼
返回结构化 JSON：
{
  "function": "graphrag_search",
  "arguments": {
    "brand": "XX茶饮",
    "city": "杭州",
    "query": "门店扩张趋势"
  }
}
    │
    ▼
Java 后端执行这个函数，拿到结果
    │
    ▼
把结果返回给大模型
    │
    ▼
大模型继续判断："我还需要调用 bocha_search 工具"
    ...（循环直到信息收集完毕）
    │
    ▼
大模型生成最终回答
```

**Java 实现关键代码逻辑：**

```java
// 伪代码，面试时能讲清楚这个流程就行
public class AgentService {

    // 注册可用工具
    private Map<String, Function> tools = Map.of(
        "graphrag_search", this::searchGraphRAG,
        "bocha_search", this::searchRealtimeNews,
        "generate_video", this::generateVideo,
        "get_similar_cases", this::getSimilarCases
    );

    public String chat(String userMessage) {
        // 1. 把问题发给大模型
        DoubaoResponse response = doubaoClient.chat(userMessage);

        // 2. 如果大模型要调用工具
        while (response.hasFunctionCall()) {
            String funcName = response.getFunctionName();
            String args = response.getFunctionArgs();

            // 3. Java 执行对应工具
            String result = tools.get(funcName).apply(args);

            // 4. 把结果返回给大模型，让它继续
            response = doubaoClient.chatWithToolResult(result);
        }

        // 5. 大模型收集完信息，生成最终回答
        return response.getContent();
    }
}
```

### 5.4 冲突消解规则（面试必问）

当实时信号和历史规律冲突时怎么办？

```
场景：历史数据显示XX品牌"预算缩减，不建议跟进"
     但今天新闻"XX品牌刚融资10亿"

冲突消解规则（Java 代码层面的规则引擎，不是交给大模型判断）：

┌─────────────────────────────────────────────────┐
│ 信号优先级：                                      │
│                                                  │
│ 实时高优信号 > 历史规律    → 覆盖历史结论          │
│   （融资/代言/政策变化）       "不建议" → "建议跟进" │
│                                                  │
│ 实时中优信号 + 历史规律    → 增强置信度            │
│   （行业活动/新品发布）        置信度 70% → 85%    │
│                                                  │
│ 实时低优信号               → 仅补充信息            │
│   （日常新闻/社交媒体）        不改变结论，只补充    │
└─────────────────────────────────────────────────┘
```

**为什么用 Java 规则引擎而不是交给大模型判断？**

```
大模型判断的问题：
- 不可控：同样输入可能给出不同结论
- 不可解释：销售不知道为什么结论变了
- 不稳定：偶尔会"幻觉"出错误判断

Java 规则引擎的优势：
- 确定性：相同输入永远相同输出
- 可解释：能明确说"因为融资信号优先级高"
- 可审计：每次决策有完整日志
```

## 六、信源可信度设计

### 6.1 博查 API 的白名单机制

```
不是什么新闻都信，只信"可信信源"：

✅ 可信信源（接入）：
- 新华社、人民日报等官媒
- 上市公司公告（巨潮资讯）
- 行业权威媒体（36氪、虎嗅）

❌ 不可信信源（过滤）：
- 自媒体公众号
- 营销号
- 社交媒体（微博/抖音）
```

### 6.2 信号过期机制

```
同一品牌24小时内同类事件只取最高信源：

5月28日 10:00  新华社报道"XX茶饮签约代言人"  → 高优 ✅
5月28日 11:00  某自媒体也报道了               → 丢弃 ❌（已有高优）
5月28日 14:00  36氪深度分析文章               → 中优，补充 ✅

5月29日（24小时后）
              原来的"代言人"信号自动降级       → 不再作为高优信号
```

## 八、总结：AI 系统的核心技术亮点

面试时用一句话总结：

> **基于已有的投前投后数据系统，调研并设计了一套"GraphRAG 智能决策 + Agent 自动编排 + 视频自动生成"的 AI 销售赋能系统。**

三个核心技术亮点：
1. **GraphRAG 替代纯向量检索**：解决了多跳推理问题（品牌→城市→指标→相似案例）
2. **Agent 替代固定流程**：大模型自己决定调用哪个工具，而非写死的 if-else
3. **规则引擎替代大模型判断**：冲突消解用 Java 代码保证确定性和可解释性

## 十、技术发散：在广告领域这个系统还可以怎么做

基于你在德高的真实业务场景，以下是合理的技术发散，可以作为面试加分项：

### 10.1 知识图谱可以加入更多维度

```
当前设计（录音中的）：
  Brand → City → StoreMetric → Campaign → Report

可扩展的节点：
  Season（季节）     → 广告有明显的季节性（夏季饮料/冬季汽车）
  Festival（节日）   → 春节/双11/618 等营销节点影响预算释放
  Competitor（竞品） → 同行业竞品的投放动作影响决策
  Regulation（政策） → 广告法/行业监管变化影响投放策略
```

### 10.2 视频生成可以结合你系统的真实数据

```
你的 generic-am-back 已经有：
  - Asset（广告位的物理尺寸/位置）
  - Traffic（人流量数据）
  - Impression（曝光量预估）

AI 系统可以用这些数据增强视频生成：
  - 根据 Asset 尺寸自动调整广告图比例
  - 根据 Traffic 数据选择最佳拍摄角度（人流量大的角度优先）
  - 根据 Impression 预估生成"效果预测"叠加在视频上
```

### 10.3 决策引擎可以引入时间序列预测

```
当前：基于历史规律 + 实时信号做判断
可扩展：引入时间序列预测（ARIMA/Prophet）

  品牌过去12个月的投放预算 → 预测下个月预算释放概率
  行业周期性规律 → 预测行业整体投放趋势
```

### 10.4 Agent 可以加入反馈学习

```
当前：Agent 给建议 → 销售执行
可扩展：销售执行结果回流 → 训练 Agent 决策模型

  Agent 建议"跟进品牌A" → 销售跟进了 → 成单了 → 正反馈
  Agent 建议"跟进品牌B" → 销售跟进了 → 没成单 → 负反馈
  → 持续优化 Agent 的置信度模型
```
