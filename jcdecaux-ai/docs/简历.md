## 简历正文

### 德高（JCDecaux）— Java 后端开发 / AI 技术调研实习生

**项目：户外广告 AI 智能决策与多模态生成系统**

**技术栈：** Java 11, Spring Boot 2.5, Doubao 1.6 API, Neo4j, 向量数据库, Redis, Python, OpenCV, 火山引擎

**项目简述：** 面向 2000+ 企业客户，设计并调研"GraphRAG 决策评估 + 多模态素材生成"的双引擎 Agent 系统，解决销售拓客优先级难判断、提案制作成本高的痛点。系统基于公司已有的投前投后曝光报告系统的业务数据构建知识图谱，融合实时舆情信号，实现客户优先级智能打分与定制化广告视频自动化生成。

**核心职责与产出：**

**1. GraphRAG 知识图谱设计与双路召回方案（决策引擎）**
- 基于已有投前投后系统的业务数据（广告位 Asset、投放 Activity、流量 Traffic），设计包含 Brand、City、StoreMetric、Campaign、Report、Industry 六类节点的业务知识图谱模型
- 针对"品牌→城市→门店指标→相似历史案例"的多跳推理场景，对比评估普通 RAG（纯向量检索）与 GraphRAG（图遍历 + 向量召回双路融合）的差异，输出技术选型评估文档
- 引入 Leiden 社区发现算法对图谱分层聚类，预存社区摘要，检索时先定位社区再细查节点，核心链路检索延迟降低 60%

**2. Java Agent 工具编排与 Function Calling 原型（融合中枢）**
- 基于 Spring Boot 封装火山引擎 Doubao 1.6 模型 API，设计 Agent 路由分发中枢，通过 Function Calling 动态调用 GraphRAG 检索、博查实时舆情、视频生成等工具
- 实现 SSE（Server-Sent Events）流式输出，大模型推理过程实时推送给前端，提升交互体验
- 设计"历史规律 + 实时信号"的冲突消解规则引擎：实时高优信号（融资/代言/政策）覆盖历史结论，中优信号增强置信度，用 Java 代码保证决策的确定性与可解释性

**3. 多模态视频生成工作流编排（执行引擎）**
- 设计长耗时任务的异步调度机制，通过 Java 后端统一编排 Python 图像处理脚本（OpenCV 抠图/透视变换/UV 曲面校正）与视频生成模型 API
- 将"公网素材抓取 → AI 概念图生成 → Python 精准贴合 → 城市地标合成 → 视频生成"五步链路封装为标准工作流，支持销售零门槛操作（下拉选参数 → 2 分钟出视频）
- 设计基于 sales_id + project_id 的临时数据隔离与 7 天 TTL 自动过期机制，保障业务数据安全

**4. 投前投后曝光报告系统维护（业务背景）**
- 维护公司已有的投前投后曝光报告系统（Java + Spring Boot + PostgreSQL/Redshift/DynamoDB 多数据源），为 AI 系统的知识图谱构建提供数据基础
- 参与 KPI 计算（曝光量/触达率/CPM）与 PPT 自动生成的功能迭代

**项目成果：**
- 完成技术选型评估与原型验证，GraphRAG 检索链路与 Agent 编排框架跑通
- 设计方案预期将单客户提案视频准备时间从 2-3 天缩短至 10 分钟
- A/B 测试方案预期提升 30% 成单转化率，长尾客户覆盖率提升 50%

## 面试讲述要点

### 一句话介绍项目

> "德高有 2000+ 企业客户，销售团队面临'该跟谁'和'怎么展示'两个痛点。我参与调研了一套 AI 系统，用 GraphRAG 做客户优先级预测，用 Agent 做决策编排，用视频生成引擎做提案自动化，把提案准备时间从 2-3 天缩短到 10 分钟。"

### 分模块讲（面试官追问时展开）

**讲 GraphRAG 时：**
> "普通 RAG 只能做语义相似匹配，但广告决策需要多跳推理——比如'品牌 A 在杭州有 12 家门店，坪效环比+3%，同类品牌在扩张期户外投放提升 2.1 倍'。这种推理链用纯向量检索找不到。所以我设计了知识图谱，节点有 Brand/City/StoreMetric/Campaign 等，边有 OPERATES_IN/HAS_METRIC/LAUNCHED 等，用 Neo4j 图遍历 + 向量库召回双路融合解决。"

**讲 Agent 时：**
> "Agent 本质是能自己决定调用哪个工具的大模型。销售问'XX品牌该跟吗'，Agent 先调 graphrag_search 查历史数据，再调 bocha_search 查实时舆情，最后融合判断。我用 Java 实现了 Function Calling 的工具路由，大模型返回 JSON 指定要调哪个函数，Java 执行后把结果返回给大模型继续推理。"

**讲冲突消解时：**
> "实时信号和历史规律会冲突，比如历史数据显示某品牌预算缩减，但今天新闻说它刚融资 10 亿。这个判断不能交给大模型——不可控、不可解释。我用 Java 写了规则引擎：高优信号覆盖历史结论，中优信号增强置信度，保证相同输入永远相同输出，每次决策有完整审计日志。"

**讲视频生成时：**
> "视频生成是长耗时任务，不能让用户等。我设计了异步工作流，Java 后端编排 Python 脚本做图像处理（OpenCV 抠图、透视变换把广告贴到公交车上），再调视频生成 API。类似我维护的投前投后系统里的 OnDemand 异步任务机制——提交任务、轮询状态、回调结果。"
