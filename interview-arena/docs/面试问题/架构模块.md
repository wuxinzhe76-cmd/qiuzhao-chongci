# 架构模块面试题

> 基于项目 `interview-arena` 的 Orchestrator + Harness 整体架构，覆盖编排模式、目标漂移防护、分层设计、模块协作等考点。

---

## 一、整体架构设计

### Q1：你的项目整体架构是怎么设计的？为什么要分层？

**答**：

六层架构，自上而下：

```
Controller（入口层）
  └── Service（薄层，委托给 Orchestrator）
        └── Orchestrator（编排层）
              ├── ToolExecutor（工具执行器：限流+权限+审计+异常兜底）
              │     └── 7 个 Tool 实现
              ├── MemoryManager（记忆管理器：短期 Redis + 长期 MySQL+Milvus）
              └── Harness 组件（InputSanitizer + OutputMonitor + LoopDetector + GoalDriftDetector）
                    └── ThreeLayerController（三层控制：AI主导+代码兜底+用户主动）
                          └── LLM 模块（ChatClient 工厂：面试/RAG/降级三个 Client）
```

分层原因：
1. **单一职责**：每层只做一件事。Tool 只管执行原子操作，Memory 只管数据存取，Harness 只管安全防护，Orchestrator 只管编排流程。
2. **可替换性**：LLM 从通义千问换成 GPT，只改 LLM 模块。Redis 挂了降级到 MySQL，只改 Memory 模块。其他层不受影响。
3. **可测试性**：ToolUnitTest 不需要 Spring Context，直接 new ToolExecutor + Mock Tool 就能测。

### Q2：为什么不用 Spring AI Alibaba 的 Agent 框架（ReactAgent）？

**答**：

之前用过 ReactAgent，后来删了，改用代码编排。原因：

1. **面试场景是 Workflow 不是 Agent**：工具调用时序是确定的（抽题 -> 评估 -> 保存 -> 路由），不需要 LLM 自主决策调用哪个工具。用代码编排比 FC 更可靠。
2. **可控性**：ReactAgent 的 ReAct 循环是 LLM 自主控制，可能死循环、可能调错工具。代码编排每一步都是确定的，三层控制保证面试不会失控。
3. **面试展示**：自己写的 Orchestrator + Tool + Harness 能讲出设计思路。用框架封装的 Agent 只能说"我用了框架"，讲不出细节。
4. **Token 成本**：ReactAgent 每轮要让 LLM 决策调哪个工具，多一次 LLM 调用。代码编排直接调工具，省一次 LLM 调用。

### Q3：Orchestrator 和 Service 的职责怎么分？

**答**：

- **Service（薄层）**：参数校验 + 权限校验 + 委托给 Orchestrator。不含业务逻辑。
- **Orchestrator**：面试流程编排。调用 Tool 执行操作，调用 Memory 管理记忆，调用 Harness 做安全防护，调用 ThreeLayerController 做路由决策。

```
Controller -> Service（校验参数+鉴权） -> Orchestrator（编排流程）
                                          ├── ToolExecutor.execute("pickQuestion")
                                          ├── MemoryManager.pushHistory()
                                          ├── InputSanitizer.sanitizeInput()
                                          └── ThreeLayerController.applyControl()
```

Service 是 Spring MVC 的标准层（@Service），Orchestrator 是业务编排层。分开的原因是 Service 可能被其他入口复用（如 MQ Consumer 直接调 Orchestrator 跳过 Service 的 HTTP 校验）。

## 二、目标漂移防护

### Q4：什么是目标漂移？你的项目怎么解决的？

**答**：

目标漂移（Goal Drift）：LLM 充当面试官时，几轮对话后可能"忘记职责"，从面试官降级为普通知识讲解器。用户利用这一点不断提问，导致面试无法推进。

解决方案：`GoalDriftDetector`，代码层拦截，不依赖 LLM 自律。

```
用户输入 -> GoalDriftDetector.checkDrift()
              │
              ├── 检测：用户在"提问"还是"回答"？
              │   ├── 疑问词开头：什么是/为什么/怎么/请问
              │   ├── 请求讲解：给我讲讲/帮我解释/详细说说
              │   └── 疑问句：5-30字 + 问号结尾
              │
              ├── 正常回答 -> 放行，继续 AI 评估
              │
              └── 检测到漂移
                  ├── 第 1 次：警告"请不要主动提问"
                  └── 第 2 次：固定话术 + 强制换题
```

### Q5：目标漂移检测为什么不用 LLM 来判断？

**答**：

三个原因：

1. **省 Token**：每次漂移检测都调 LLM 会多一次 API 调用。用正则匹配零成本。
2. **速度**：正则匹配 <1ms，LLM 调用 2-5 秒。漂移检测在热路径上，不能慢。
3. **可靠性**：LLM 自己可能已经漂移了（这就是问题所在），让一个已经漂移的 LLM 判断是否漂移，不可靠。代码层正则是确定性的。

### Q6：漂移对话为什么不写入对话历史？

**答**：

```java
// 检测到漂移时：
// ✅ 保存到 MySQL（面试记录完整性）
// ❌ 不写入 Redis 对话历史（防止 LLM 被带偏）
```

原因：Redis 对话历史会拼到 Prompt 里喂给 LLM。如果漂移对话（如"什么是 HashMap？"）进入历史，LLM 下一轮会看到这个提问，可能"顺着"用户的问题继续讲解，加剧漂移。不写入历史 = LLM 永远看不到这次漂移对话，不会被带偏。

### Q7：目标漂移和三层控制是什么关系？

**答**：

两者都是 Harness L5 熵管理的防护机制，但层次不同：

| 机制 | 触发条件 | 执行层 | 作用 |
|------|---------|--------|------|
| GoalDriftDetector | 用户在"提问"而非"回答" | Orchestrator（LLM 调用前） | 防止 LLM 被带偏 |
| ThreeLayerController 兜底1 | 单题追问 >3 轮 | Orchestrator（LLM 调用后） | 防止 AI 在一道题上无限追问 |
| ThreeLayerController 兜底2 | 总轮次 >=10 | Orchestrator（LLM 调用后） | 防止 AI 永远不结束 |

GoalDriftDetector 是**输入侧防护**（在 LLM 看到输入之前拦截），ThreeLayerController 是**输出侧防护**（在 LLM 返回结果后覆盖决策）。两者互补：前者防用户带偏 LLM，后者防 LLM 自己跑偏。

## 三、Harness 工程化

### Q8：你的 Harness 层做了哪些防护？在代码的什么位置？

**答**：

按调用链顺序，7 个 Harness 防护点：

| 序号 | 防护 | Harness 层 | 位置 | 作用 |
|------|------|-----------|------|------|
| ① | InputSanitizer | L3 安全 | Orchestrator 步骤 0 | 防注入：12 种攻击模式清洗 |
| ② | GoalDriftDetector | L5 熵 | Orchestrator 步骤 0.5 | 目标漂移：防止用户提问带偏 LLM |
| ③ | LoopDetector | L5 熵 | Orchestrator 步骤 3 | 循环检测：连续相同/Ping-Pong/最大轮次 |
| ④ | Sentinel 限流 | L2 治理 | ToolExecutor 入口 | QPS 限流：防高并发打 MySQL |
| ⑤ | ToolPermission | L3 安全 | ToolExecutor 权限检查 | READ/WRITE/CRITICAL 三级权限 |
| ⑥ | CircuitBreaker | L2 治理 | AbstractLlmTool 内部 | 熔断器：LLM 故障快速失败 |
| ⑦ | OutputMonitor | L3 安全 | Orchestrator 步骤 6 | 输出监控：长度/重复/异常模式检测 |

### Q9：为什么 LLM 工具的 Harness 防护比普通工具多？

**答**：

LLM 调用有四个特殊性，需要额外防护：

1. **有成本**：按 Token 计费 -> 需要 TokenBudget 预算控制
2. **可能超时**：API 响应慢 -> 需要 CircuitBreaker 熔断快速失败
3. **可能限流**：通义千问 429 -> 需要 StructuredErrorHandler 分类 + 重试决策
4. **输出不可控**：可能幻觉/异常 -> 需要 OutputMonitor 监控

普通工具（如 PickQuestion 查 MySQL）没有这些问题：无成本、毫秒级响应、输出确定。所以只需要限流 + 权限 + 异常兜底就够了。

### Q10：CircuitBreaker 和 Sentinel 限流有什么区别？为什么要两个？

**答**：

| 维度 | Sentinel 限流 | CircuitBreaker 熔断 |
|------|-------------|-------------------|
| 目的 | 控制流量（防过载） | 防止级联故障（防雪崩） |
| 触发 | QPS 超阈值 | 失败次数超阈值 |
| 位置 | ToolExecutor（所有工具） | AbstractLlmTool（仅 LLM 工具） |
| 恢复 | QPS 下降自动恢复 | 冷却 60s -> 半开试探 |

两者互补：Sentinel 防止外部高并发打进来（入口限流），CircuitBreaker 防止内部 LLM 故障扩散出去（出口熔断）。比如双 11 大促时 Sentinel 限制用户请求量，通义千问宕机时 CircuitBreaker 快速失败走兜底。

## 四、编排模式

### Q11：你的 Orchestrator 是什么编排模式？和 ReAct 有什么区别？

**答**：

我的 Orchestrator 是**代码编排（Workflow）模式**，不是 ReAct（推理-行动循环）模式。

| 维度 | ReAct（已删除） | 代码编排（当前） |
|------|----------------|----------------|
| 决策者 | LLM 自主决策调哪个工具 | 代码固定流程 |
| 灵活性 | 高（LLM 可自主探索） | 低（流程确定） |
| 可控性 | 低（可能死循环/调错工具） | 高（每步确定） |
| Token 成本 | 高（每轮多一次决策调用） | 低（直接调工具） |
| 适用场景 | 开放式探索（如数据分析） | 固定流程（如面试） |

面试场景的流程是确定的：开始 -> 抽题 -> 评估 -> 路由 -> 结束。不需要 LLM 决策"下一步做什么"，用代码编排更可靠。

### Q12：三层控制的核心思想是什么？

**答**：

核心思想：**AI 主导决策，但代码做兜底**。

```
AI 返回 action_directive（DEEP_DIVE / NEXT_QUESTION / END_INTERVIEW）
      │
      ├── 代码兜底 1：单题 >3 轮？ -> 强制 NEXT_QUESTION
      ├── 代码兜底 2：总轮次 >=10？ -> 强制 END_INTERVIEW
      └── 最终执行 AI 指令（或代码覆盖后的指令）
```

为什么需要代码兜底：
1. AI 可能在一道题上无限追问（不敢给 NEXT_QUESTION 怕"不礼貌"）-> 代码强制换题
2. AI 可能永远不结束面试（觉得用户"还有进步空间"）-> 代码强制结束
3. AI 的判断不可 100% 信任 -> 代码做安全网

这是"**信任但验证**"原则：信任 AI 的决策能力，但用代码验证边界条件。

## 五、异步与降级

### Q13：面试报告生成为什么走 MQ 异步？

**答**：

报告生成涉及 LLM 调用（10-30s）+ PDF 生成（10-30s），总耗时 20-60 秒。

```
用户调 POST /end -> GenerateReportTool 发 MQ -> 立即返回"面试已结束"
                      │
                      ▼
                MQ Consumer 异步
                  ├── 记忆整合
                  ├── LLM 生成评估
                  └── MCP 生成 PDF
```

同步的话用户要等 60 秒，体验差。MQ 异步的优势：
- **可靠投递**：消息持久化，进程崩了不丢任务
- **解耦**：生产者和消费者分离
- **手动 ACK**：消费失败可选择重试或丢弃

### Q14：PdfReportService 超时控制是怎么做的？为什么不用 @Async？

**答**：

用 `CompletableFuture.supplyAsync().or(60, TimeUnit.SECONDS).join()`：

```java
String result = CompletableFuture
    .supplyAsync(() -> doGenerateReportPdf(sessionId))
    .or(60, TimeUnit.SECONDS)  // 60秒超时
    .join();
```

不用 `@Async` 的原因：
1. `@Async` 没有超时控制，任务可能无限阻塞线程池
2. `@Async` 的异常处理复杂（需要 AsyncUncaughtExceptionHandler）
3. `CompletableFuture.or()` 超时后能 catch CancellationException 走降级逻辑

超时降级：不调 LLM（可能就是 LLM 导致的超时），只从 MySQL 拉对话记录拼纯文本 Markdown。

### Q15：你的项目有哪些降级方案？

**答**：

四层降级：

| 场景 | 降级方案 | 实现 |
|------|---------|------|
| LLM 调用失败 | 返回兜底 DTO（END_INTERVIEW） | AbstractLlmTool.fallbackResponse() |
| 主模型故障 | 降级到备用 ChatClient | FallbackChain（RagConfig 注册） |
| PDF 生成超时 | 降级为保存 Markdown 文件 | PdfReportService.saveMarkdownFallback() |
| 目标漂移 | 跳过 LLM 调用，返回固定话术 | GoalDriftDetector.checkDrift() |

降级原则：**不影响主流程**。LLM 挂了面试能结束（兜底 DTO），PDF 挂了至少有 Markdown，用户提问带偏了能被拦截。

## 六、模块协作

### Q16：Tool、Memory、Harness 三个模块是怎么协作的？

**答**：

以 `answerInterview()` 的一次完整调用为例：

```
Orchestrator.answerInterview()
  │
  ├── InputSanitizer.sanitizeInput()           ← Harness：防注入
  │
  ├── GoalDriftDetector.checkDrift()           ← Harness：目标漂移检测
  │     └── StringRedisTemplate（违规计数存 Redis）
  │
  ├── ToolExecutor.execute("saveRecord")       ← Tool：保存记录
  │     ├── Sentinel 限流检查                    ← Harness：QPS 控制
  │     ├── ToolPermission 权限检查              ← Harness：WRITE 审计
  │     └── SaveRecordTool -> InterviewRecordMapper（写 MySQL）
  │
  ├── MemoryManager.pushHistory()              ← Memory：写对话历史（Redis）
  │
  ├── LoopDetector.detectLoop()                ← Harness：循环检测
  │
  ├── ToolExecutor.execute("getQuestionDetail") ← Tool：取题目（Redis 缓存）
  │
  ├── MemoryManager.getHistory()               ← Memory：取最近 10 条（Redis）
  │
  ├── ToolExecutor.execute("evaluateAnswer")   ← Tool：AI 评估
  │     ├── Sentinel 限流（10 QPS）
  │     └── AbstractLlmTool.callLlm()
  │           ├── TokenBudget.checkBudget()     ← Harness：Token 预算
  │           ├── CircuitBreaker.executeProtected() ← Harness：熔断器
  │           └── ChatClient.call().entity(DTO)  ← LLM：通义千问
  │
  ├── OutputMonitor.monitor()                  ← Harness：输出监控
  │
  └── ThreeLayerController.applyControl()       ← Harness：三层控制
        └── DEEP_DIVE / NEXT_QUESTION / END_INTERVIEW
```

关键协作点：
1. **Tool 执行前后被 Harness 包裹**：ToolExecutor 在 Tool.execute() 前后加限流+权限+审计+异常兜底
2. **Memory 被 Tool 和 Orchestrator 共享**：Tool（如 SaveRecordTool）写 MySQL，Orchestrator 通过 MemoryManager 读写 Redis
3. **Harness 穿插在调用链的每个关键节点**：不是集中式拦截，而是分布式穿插（防注入在入口、限流在 Tool 层、熔断在 LLM 层、监控在出口）

### Q17：如果要新增一个"学习建议"功能，需要改哪些模块？

**答**：

以"面试结束后基于薄弱点生成学习建议"为例：

| 模块 | 改动 | 说明 |
|------|------|------|
| LLM | ❌ 不改 | 已有 ChatClient，不需要新模型 |
| Tool | ✅ 新增 GenerateAdviceTool | extends AbstractLlmTool，拼 Prompt -> callLlm() |
| ToolExecutor | ❌ 不改 | 自动注册（@Component），自动限流（SentinelConfig 加规则） |
| Memory | ❌ 不改 | 已有 getPersistentWeakPoints() 取薄弱点 |
| Harness | ❌ 不改 | AbstractLlmTool 自动提供 CircuitBreaker+TokenBudget |
| Orchestrator | ✅ 在 doEndInterview() 加一行 | toolExecutor.execute("generateAdvice", ...) |

只需改 2 个模块（新增 1 个 Tool + Orchestrator 加 1 行），其他模块零改动。这就是分层架构的价值：**扩展点集中，改动范围可控**。
